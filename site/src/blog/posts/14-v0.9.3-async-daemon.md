The goal at the end of v0.9.2 was clear: agents spend too much time bootstrapping context. Every `synlynk exec` re-reads project-docs, regenerates context.md, and hands it to the agent — that's the right model for one-off invocations, but it's friction for agents that need to check context frequently or queue work without a human in the loop.

v0.9.3 removes that friction. synlynk can now run as a daemon.

## What shipped

### `synlynk daemon`

```
synlynk daemon start               # double-fork, write .synlynk/daemon.pid
synlynk daemon stop
synlynk daemon status              # pid · uptime · queue depth · HTTP URL
synlynk daemon restart
synlynk daemon --install-service   # launchd / systemd / @reboot crontab
synlynk daemon --uninstall-service
```

The daemon is a subclass of the existing `WatchDaemon` — it inherits the double-fork daemonization, mtime polling on `project-docs/`, and `context.md` regeneration. What it adds is a second thread: an HTTP server on `localhost:27471`.

### HTTP API

Ten endpoints, all stdlib (`http.server`), all localhost-only:

| Method | Path | What it does |
|--------|------|--------------|
| `GET` | `/context` | Current `context.md` as JSON or plain text |
| `GET` | `/status` | Uptime, pid, queue counts |
| `GET` | `/jobs` | All jobs; filterable with `?status=` |
| `GET` | `/jobs/<id>` | Single job + last 100 lines of log |
| `POST` | `/dispatch` | Enqueue a job with agent, task, priority, depends_on |
| `GET` | `/stories` | Story list from state.db |
| `GET` | `/stories/<id>` | Single story |
| `GET` | `/capability` | Capability ratings by agent × domain |
| `GET` | `/sentinel` | Active sentinel alerts |
| `POST` | `/checkpoint` | Force-regenerate context.md immediately |

Agents can now pull context with a single `curl` rather than a full CLI invocation. The daemon keeps it fresh automatically.

### Persistent job queue

The new `daemon_jobs` table in `state.db` survives restarts:

```sql
CREATE TABLE IF NOT EXISTS daemon_jobs (
    job_id       TEXT PRIMARY KEY,
    agent        TEXT NOT NULL,
    task         TEXT NOT NULL,
    status       TEXT NOT NULL DEFAULT 'queued',   -- queued → running → done | failed
    priority     INTEGER NOT NULL DEFAULT 5,        -- 1 (highest) to 10 (lowest)
    depends_on   TEXT NOT NULL DEFAULT '[]',        -- JSON array of job_ids
    pid          INTEGER,
    enqueued_at  TEXT NOT NULL,
    ...
);
```

Three correctness properties that went into the implementation:

**Zombie-safe reaping.** `os.kill(pid, 0)` returns `True` for zombie processes — child processes that have exited but haven't been reaped. The original implementation would have treated every finished job as still running. Fixed with `os.waitpid(pid, os.WNOHANG)`, with a `ChildProcessError` fallback for processes adopted by init after a daemon restart.

**Dependency failure propagation.** If a job's dependency fails, a naïve "wait until all deps are done" check would leave the downstream job queued forever. The dispatcher now checks for `failed` deps explicitly and immediately marks the blocked job failed rather than deadlocking.

**Per-job commit.** The original implementation batched all dispatched jobs in a single commit at the end of the loop. A crash between spawning a process and committing would leave the job as `queued` in the DB while the process was already running — producing a duplicate on restart. Now each job is committed immediately after its process is spawned.

### Service installation

```
synlynk daemon --install-service
```

Detects platform and writes the appropriate unit:

- **macOS:** `~/Library/LaunchAgents/com.synlynk.daemon.plist` → `launchctl load -w`
- **Linux:** `~/.config/systemd/user/synlynk-daemon.service` → `systemctl --user enable --now`
- **Fallback:** `@reboot synlynk daemon start` appended to crontab

## How it was built

This was the first v0.9.x feature to use the three-agent authoring protocol in full:

- **Claude** (Tasks 1–3): daemon_jobs schema, `_reconcile_daemon_jobs`, `_dispatch_ready_jobs`, `SynlynkDaemon` class, all 10 HTTP endpoints — PR #56
- **Agy** (Task 4): `synlynk daemon` CLI parser and dispatch block in `main()` — PR #57
- **Codex** (Tasks 5–6): `--install-service`/`--uninstall-service` platform implementations + VERSION bump to 0.9.3 — PR #58

Reviews were cross-agent (non-author reviews) and fixes were author-only. Agy found the zombie process bug and the per-job commit race in the spec compliance review of PR #56 — both were fixed before merge. Codex's review of PR #57 caught missing CLI path tests. Claude's review of PR #58 caught the log path using `~/synlynk/` instead of `~/.synlynk/`.

## What's next

The daemon exposes the job queue over HTTP, but v0.9.3 still requires the human (or another CLI call) to enqueue work. The next natural step is **v0.9.4 Workgroup Relay** — a WSS relay that lets agents on different machines share context and dispatch across a workgroup, without requiring a shared filesystem.

The daemon's HTTP API is the foundation that makes that possible.
