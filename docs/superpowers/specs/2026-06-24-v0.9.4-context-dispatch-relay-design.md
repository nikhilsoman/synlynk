## 1. Context Mode and Scoping

### Decision (Q1, Q6, Q7)

`dispatch_agent()` accepts a `--context-mode` flag with three values:

| Mode | Context injected | When to use |
|------|-----------------|-------------|
| `none` | empty string | self-contained tasks (cross-repo PR review, one-shot instructions) |
| `task` | story + active tasks + scoped architecture (~400–2000 bytes) | default — most dispatches |
| `full` | current `generate_context()` output (~80–120KB) | explicit request only |

**Default:** `task`. All new dispatches are `task` unless the user passes `--context-mode full` or the agent profile overrides it.

**Self-contained vs context-needed distinction:** the 119KB rxcc PR review dispatch was a self-contained task — its context requirement was zero (task string was fully specified). The cross-repo framing was incidental. `--context-mode none` is the correct flag for any task where the `--task` string is fully self-describing.

### Mandatory fields for `task` mode (Q6)

`_generate_task_context(story_id)` must include:
- story title + description + current status
- active subtasks for that story
- agent name + capability framing line (from `.agents/<agent>.json` if present)
- scoped source architecture (existing behaviour, kept)

Optional / suppressible:
- teammates' devlog entries (default: off in `task` mode)
- full roadmap context (default: off in `task` mode)

## 2. Context Delivery — In-Memory (Q8)

### Decision

`generate_context()` returns the context string as its return value. The dispatch path uses the return value directly — no file write in the dispatch path.

**Before (broken):**
```python
generate_context(scope)          # writes .synlynk/context.md
context = open("context.md").read()  # race: may have been overwritten
```

**After (fixed):**
```python
context_text = generate_context(scope)   # returns string, no file write
_format_prompt_for_agent(context_text)   # uses return value directly
```

**Daemon path unchanged:** the HTTP endpoint path still writes `.synlynk/context.md` via the existing lock-guarded path. The fix is purely in `dispatch_agent()`.

**Audit trail:** `.synlynk/prompts/<job_id>.md` already exists per-job and contains the full assembled prompt including injected context. This is the de facto per-job context record. No separate `jobs/<job_id>/context.md` needed (Option C rejected as over-engineering).

## 3. Context Size Budget (Q9)

Two thresholds, both configurable in `.synlynk/config.json`:

| Key | Default | Behaviour |
|-----|---------|-----------|
| `context_warn_bytes` | `81920` (80KB) | warn in dispatch output, continue |
| `context_tokens_max` | `null` (disabled) | if set, blocks dispatch and shows escape hatch |

Warning output (soft):
```
⚠ context: full (94KB) — exceeds soft limit (80KB)
  Use --context-mode task to reduce to ~820 bytes
```

Blocked output (hard, when `context_tokens_max` set):
```
✗ context: full (119KB) — hard limit is 100KB
  --context-mode task / --context-mode none / --override-context-limit
```

Context size is shown inline in the dispatch header line for all modes:
```
◎ job 7f3a · claude · context: task (820 bytes)
```

## 4. Per-Agent Context Profiles (Q10)

### File location

`.agents/<agent>.json` — optional override files per agent, checked into the repo.

### Schema

```json
{
  "agent": "agy",
  "context_mode": "task",
  "context_max_bytes": 2000,
  "instruction_file": ".agents/agy-instructions.md",
  "capability_tags": ["code-review", "diff-analysis"],
  "context_truncation": "tail"
}
```

All fields optional. `dispatch_agent()` merges agent profile over dispatch defaults at runtime.

**Agy migration:** the current hardcoded `context_text[:2000]` in `_format_prompt_for_agent()` becomes `context_max_bytes: 2000` in `.agents/agy.json`. The truncation logic moves to a shared helper.

### Creation

`synlynk agent configure <agent>` — interactive prompt that writes `.agents/<agent>.json`. Or hand-edited.

## 5. SQLite Canon (DevX-2)

**Primary store:** `stories` table in `state.db` is canonical for task state from v0.9.4 onwards.

**todo.md:** generated view. Daemon writes it on mutation; human edits to todo.md are imported by daemon on next start as one-time migration (`- [ ]` items → new story rows with `status=queued`).

**No dual-write:** the daemon does not maintain todo.md as a live mirror. It writes it once after each mutation for grep/editor access. The source of truth is SQLite.

## 6. Relay Wire Protocol

### Topology

Stateless fan-out broker. Each node has its own `state.db`. The relay carries typed events between `SynlynkDaemon` instances — it holds no state. Nodes that are offline miss events; resync via snapshot pull is v0.10.0.

### Event Schema

Seven event types:

| Type | Purpose |
|------|---------|
| `story_updated` | Story status transitions |
| `job_dispatched` | Agent kicked off |
| `job_completed` | Job exit — tokens, cost, compliance tags |
| `alert_raised` | Sentinel fires — propagates to all workgroup nodes |
| `context_checkpoint` | Context regenerated — remote nodes can pull |
| `table_changed` | Generic fallback for any SQLite mutation not covered above |
| `broadcast` | Social/ambient layer — MOTD, wellness, team messages, jokes |

All events share base fields: `type`, `ts` (Unix timestamp), `origin_node` (machine ID).

### `broadcast` event

```json
{
  "type": "broadcast",
  "kind": "motd | wellness | message | joke | custom",
  "body": "🚶 Stand up. You've been heads-down for 90 minutes.",
  "from": "pm-agent@machine-a",
  "ts": 1719187200,
  "origin_node": "machine-a"
}
```

Any node or agent may publish a broadcast. CLI: `synlynk relay broadcast "<body>"`.

### Deployment staircase

| Mode | Config | Notes |
|------|--------|-------|
| LAN (mDNS) | `relay_mode: lan` | zero config, same network only |
| Cloudflare Tunnel | `relay_mode: cf-tunnel` | **default** — `synlynk relay tunnel start` |
| Fly.io VPS | `relay_mode: vps`, `relay_url: <fly-url>` | ~$3–5/mo, always-on, 35+ regions |
| Community server | v1.1 | namespaced workgroups, synlynk.com hosted |

Migration between modes = change one URL. All modes use the same WSS event protocol.

## 7. Dispatch Lifecycle UX (DevX-3, Cluster A)

### Pre-flight gate

Runs before any subprocess is spawned. Checks (in order):
1. Daemon alive (`localhost:27471`)
2. Agent installed on this machine
3. Agent not in cooldown (post-FLATLINE backoff)
4. Budget headroom (`context_warn_bytes` / `context_tokens_max`)

Silent on pass (except `--verbose`). Actionable error with fix command on any failure.

### Lifecycle output

**Background (default):**
```
✓ queued — job 7f3a · monitor: synlynk jobs
```

**Foreground (`--wait`):**
```
◎ job 7f3a · claude · pr-review · context: task (820 bytes)
▸ agent running (42s)
✓ done (2m 13s) · tokens: 1,247 in / 420 out · ~$0.009
  Output: .synlynk/jobs/7f3a/output.md
```

### `synlynk jobs` command

One-shot table. Columns: ID, AGENT, STORY, STATUS, AGE, COST. `--watch` flag streams updates. **HUD (TUI jobs board):** v1.0.

### Sentinel extension (AB-4)

New sentinel pattern: `VERIFY_SKIP` — fires when `compliance_tags` field in `job_completed` output is missing or shows `verify_before_commit: false`. Informational for v0.9.4 (no block). Builds the adherence dataset for AB epic.

`alert_raised` relay event propagates sentinel alerts to all workgroup nodes.

## 8. Open Questions (deferred post-v0.9.4)

- **Q (offline sync):** When a node reconnects after being offline, how does it resync missed `table_changed` events? → v0.10.0 snapshot protocol.
- **Q (broadcast scheduling):** Should the daemon support cron-style wellness broadcast schedule (e.g., every 90 min during active session)? → defer to PM Agent autopilot config.
- **Q (per-agent profile versioning):** If `.agents/agy.json` changes mid-session, does the running daemon pick it up? → file-watch or reload-on-dispatch; decide in OB-9.
