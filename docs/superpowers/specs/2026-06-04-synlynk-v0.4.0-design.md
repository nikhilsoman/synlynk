## 1. Context and Scope

### Why v0.4.0 exists

v0.3.0 shipped enriched agent instruction templates and `synlynk init` flags. It cannot be safely applied to rxcc because:

1. rxcc's existing `CLAUDE.md` is richer than synlynk's generic template — naive migration would destroy hand-crafted content
2. rxcc's project docs may live outside `project-docs/` — no migration path exists yet
3. `synlynk start` (the command that automates the agent handoff loop) is not yet implemented

Both features must ship together — neither is useful in isolation for rxcc.

### What v0.4.0 delivers

| Feature | Command | Value |
|---|---|---|
| Repo migration | `synlynk migrate` | Safe bootstrap of any repo, including evolved ones |
| Autonomy driver | `synlynk start <issue-id>` | Read issue → move board → inject context → launch agent |

### Explicitly deferred to v0.5.0

- Textual TUI (5 core panels)
- SQLite rooms / WIP signals
- Context Server / room subscriptions
- Competency tracking

## 3. `synlynk migrate`

### 3.1 CLI

```bash
synlynk migrate            # scan, detect maturity, prompt, execute
synlynk migrate --dry-run  # show what would happen, no writes
```

`synlynk init` output gains a nudge when project docs are detected outside `project-docs/`:

```
  ⚠ Existing docs detected outside project-docs/ — run `synlynk migrate` to move them.
```

### 3.2 Step 1 — Scan

Walk the repo tree (excluding `.git/`, `node_modules/`, `.synlynk/`). Flag:

**Project doc candidates** — files whose name matches any of:
`roadmap.md`, `todo.md`, `memory.md`, `costs.md`, `devlog.md`, plus any directory named `devlogs/`
…that live **outside** `project-docs/`.

**Agent file candidates** — files at repo root matching:
`CLAUDE.md`, `GEMINI.md`, `AI_INSTRUCTIONS.md`, `AGENTS.md`
…that have substantial existing content (see maturity check below).

### 3.3 Step 2 — Maturity Check

For each agent instruction file found:

**Fresh** (use normal init flow):
- File absent, OR
- File ≤ 30 lines, OR
- All sections match synlynk template headers

**Evolved** (show 3-option menu):
- File > 100 lines, OR
- 3+ section headers not present in synlynk's template

### 3.4 Step 3 — Evolved Repo Menu

```
synlynk migrate — rxcc (Dialify/rxcc)

Found evolved agent instructions (CLAUDE.md: 284 lines, 7 sections).
Your existing content is richer than synlynk's generic template.

[A] Adopt + combine — preserve your content, add only what's missing
[B] Exit           — no changes (re-run migrate when ready)
[C] Full replace   — overwrite with synlynk templates (destructive)

Choice [A/b/c]:
```

**Option C requires confirmation:** user must type `replace` exactly. Warn that git is the only rollback.

### 3.5 Option A — Adopt + Combine

For each agent file, run semantic section matching (see §3.6). Then:

1. For each synlynk-owned section **not covered**: append to end of file
2. For each synlynk-owned section **covered by existing content**: skip, log as `✓ covered`
3. Extract GH Projects v2 IDs from existing file content into `.synlynk/config.json` (see §3.7)
4. Create `AGENTS.md` if absent
5. Auto-detect and write `owner`/`repo` to `.synlynk/config.json`

Show a preview before writing:

```
Analysing CLAUDE.md...
✓ covered   Live Issues SOP → your "### Live Issues" (keeping yours)
✓ covered   Anti-Amnesia → your "### Mid-Session Persistence" (keeping yours)
✓ covered   4-Doc Discipline → your "## Mandatory Document Maintenance" (keeping yours)
✓ extracted GH Projects v2 IDs → .synlynk/config.json
+ appending ## synlynk Start (not present)

✓ creating  AGENTS.md
✓ detected  owner=Dialify repo=rxcc → .synlynk/config.json

CLAUDE.md: 1 section appended, 0 replaced, 0 deleted
Proceed? [Y/n]
```

### 3.6 Semantic Section Matching

synlynk maintains a `SECTION_SIGNALS` map. For each synlynk-owned section, a set of keywords that indicate the concept is already covered — regardless of the exact header used.

```python
SECTION_SIGNALS = {
    "## Live Issues SOP": [
        "live issue", "live-issue", "sev1", "sev2", "sev3", "rca", "[live-"
    ],
    "## Mid-Session Anti-Amnesia Protocol": [
        "25,000 tokens", "25k tokens", "compaction", "compaction imminent",
        "mid-session", "checkpoint every"
    ],
    "## Mandatory 4-Doc Discipline": [
        "roadmap.md", "devlog", "costs.md", "memory.md",
        "mandatory document", "four doc", "4-doc"
    ],
    "## GitHub Projects v2 Integration": [
        "updateProjectV2", "projectId", "PVT_", "PVTSSF_",
        "github projects", "programme board"
    ],
    "## Git Worktree-First Policy": [
        "git worktree", "worktree add", "never commit to main",
        "never commit to master"
    ],
}
```

**Matching rule:** scan all existing section headers and the first 20 lines of each existing section body. If **2 or more** signals from a synlynk section's list are found anywhere in the file → section is "covered". Covered sections are not appended.

### 3.7 GH Projects v2 ID Extraction

When scanning an existing agent file, look for the pattern `PVT_[A-Za-z0-9]+` (project node ID) and `PVTSSF_[A-Za-z0-9]+` (field node IDs). Extract:

- First `PVT_` match → `project_id`
- Field IDs and option IDs: look for GraphQL mutation blocks containing `fieldId` and `singleSelectOptionId` values

Write extracted values to `.synlynk/config.json`. If extraction is ambiguous or finds multiple project IDs, prompt the user to confirm which one.

### 3.8 Project Doc Migration (both fresh and evolved)

For each project doc candidate found outside `project-docs/`:

1. If `project-docs/<filename>` does not exist: move file (rename), print `✓ moved ./roadmap.md → project-docs/roadmap.md`
2. If `project-docs/<filename>` already exists: print `⚠ project-docs/roadmap.md already exists — skipping ./roadmap.md (manual merge needed)`
3. Migrate `devlogs/` directory contents recursively if found outside `project-docs/devlogs/`

Never overwrites existing `project-docs/` files without `--force`.

## 4. `synlynk start <issue-id>`

### 4.1 CLI

```bash
synlynk start <issue-id>           # full flow
synlynk start <issue-id> --dry-run # print plan, don't move board or launch agent
```

### 4.2 Step 1 — Read Issue

```bash
gh api repos/{owner}/{repo}/issues/{issue_id}
```

Returns: `title`, `body`, `labels[].name`. Fails fast if `gh` is not authenticated or `owner`/`repo` missing from config.

### 4.3 Step 2 — Infer Agent

Scan labels for `agent:<name>` pattern. Strip prefix → `name`.

Look up `name` in agent slot config (`.synlynk/config.json` → `agent_slots`):

```json
{
  "agent_slots": {
    "claude": "claude",
    "agy":    "gemini",
    "codex":  "codex"
  }
}
```

`agent_slots` is written by `synlynk init` based on `--agents` flag and maps agent names to their CLI exec commands. If no `agent:` label found: prompt interactively — `"No agent label on issue #N. Which agent? [claude/agy/codex]"`.

### 4.4 Step 3 — Board Field Discovery

Check `.synlynk/board-cache.json`:
- Exists and `cached_at` within 7 days → use cache
- Missing or stale → run GraphQL discovery query

**Discovery query:**

```graphql
query DiscoverFields($projectId: ID!) {
  node(id: $projectId) {
    ... on ProjectV2 {
      fields(first: 20) {
        nodes {
          ... on ProjectV2SingleSelectField {
            id
            name
            options { id name }
          }
        }
      }
    }
  }
}
```

Match fields by name (case-insensitive): field named `"Status"` → `status_field_id`, field named `"Agent"` → `agent_field_id`. Match options by name to build `status_options` and `agent_options` maps.

**`board-cache.json` schema:**

```json
{
  "cached_at": "2026-06-04T10:00:00",
  "project_id": "PVT_kwDOAAUx684BYYIM",
  "status_field_id": "PVTSSF_lADOAAUx684BYYIMzhTeaz0",
  "agent_field_id": "PVTSSF_lADOAAUx684BYYIMzhTecpE",
  "status_options": {
    "Todo": "...", "In Progress": "47fc9ee4", "In Review": "...", "Done": "..."
  },
  "agent_options": {
    "claude": "ecf901a8", "agy": "...", "codex": "..."
  }
}
```

### 4.5 Step 4 — Move Board

Two GraphQL mutations (reuse existing `gh api graphql` pattern from rxcc CLAUDE.md):

1. Set Status field → `"In Progress"` option ID
2. Set Agent field → option ID matching the inferred agent name (case-insensitive match against `agent_options` keys)

Get the issue's project item ID first:

```graphql
query GetItemId($issueUrl: URI!) {
  resource(url: $issueUrl) {
    ... on Issue {
      projectItems(first: 1) { nodes { id } }
    }
  }
}
```

### 4.6 Step 5 — Build Context

1. Run `generate_context()` → writes `.synlynk/context.md`
2. Prepend an **Active Issue block** to `context.md`:

```markdown
# Active Issue: #42 — Implement FHIR patient upload endpoint

**Labels:** domain:backend, agent:claude
**URL:** https://github.com/Dialify/rxcc/issues/42

<issue body verbatim>

---

```

The rest of `context.md` follows (active tasks, roadmap, memory, devlog).

### 4.7 Step 6 — Launch Agent

```python
exec_command([agent_exec])
```

Uses existing `exec_command()` — inherits TTY, telemetry logging, flatline detection, budget check.

### `.synlynk/config.json` — new fields

```json
{
  "schema_version": 1,
  "budget": { "limit_usd": 10.0, "limit_requests": 100 },
  "watch_interval_seconds": 30,
  "org": "Dialify",
  "owner": "Dialify",

  "agent_slots": {
    "claude": "claude",
    "agy": "gemini",
    "codex": "codex"
  },
  "team": null,
  "sync_endpoint": null
}
```

New fields: `owner`, `repo`, `agent_slots`. `project_id` was already present from v0.3.0.

### `.synlynk/board-cache.json` — new file

Schema defined in §4.4. Written on first successful board field discovery. Invalidated after 7 days (TTL). Deleted and re-fetched on `synlynk migrate` (fresh discovery for the current project).

## 6. Error Handling

| Failure | Response |
|---|---|
| No git remote | Warn: "Could not detect owner/repo from git remote. Set manually in .synlynk/config.json." Continue. |
| `project_id` missing from config | `synlynk start` fails fast: "Run `synlynk init --project-id <id>` or `synlynk migrate` first." |
| No `agent:` label on issue | Prompt: "No agent label on #N. Which agent? [claude/agy/codex]" |
| Board field discovery fails | Warn: "Could not discover board fields. Skipping board move." Continue to context + launch. |
| Agent exec not found | Fail: "Agent exec '{exec}' not found. Is it installed and in PATH?" |
| `migrate --dry-run` | Print full plan with `[DRY RUN]` prefix on each action. No writes. Exit 0. |
| Option C without 'replace' typed | Abort with: "Confirmation not received. No changes made." |

---

## 7. Migration Path for Existing Installations

`synlynk migrate` is idempotent. Re-running it:
- Re-scans for any new docs outside `project-docs/`
- Re-checks for missing synlynk sections
- Skips already-covered sections
- Refreshes `board-cache.json` (forces re-discovery)
- Does not re-extract IDs already present in `config.json`

## 8. Testing

Key test scenarios:

| Scenario | Test |
|---|---|
| Fresh repo (no CLAUDE.md) | `migrate` runs normal init flow |
| Evolved repo detected | Agent file > 100 lines → 3-option menu shown |
| Option A — section covered | SECTION_SIGNALS match → section skipped, not appended |
| Option A — section missing | Section not matched → appended to file |
| GH ID extraction | `PVT_` + `PVTSSF_` patterns extracted to config |
| Option C without confirm | Abort, no writes |
| `--dry-run` | No files written, output shows `[DRY RUN]` lines |
| `synlynk start` — no agent label | Prompt shown |
| `synlynk start` — board cache valid | GraphQL discovery skipped |
| `synlynk start` — board cache stale | GraphQL discovery runs, cache refreshed |
| Remote URL parsing | `https://` and `git@` formats both produce correct owner/repo |
| `synlynk start --dry-run` | Plan printed, no board move, no agent launch |
