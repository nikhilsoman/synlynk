# v0.9.3 Async Daemon Design

**Date:** 2026-06-23  
**Version target:** v0.9.3  
**Status:** Approved for implementation

---

## Goal

Make synlynk always-running so agents can pull context without invoking the CLI each time. The daemon keeps `context.md` current via mtime polling, exposes it over a local HTTP API, and manages a persistent job queue with priority and dependency support.

---

## Architecture Overview

`SynlynkDaemon` subclasses `WatchDaemon` and adds:
1. An HTTP server thread on `localhost:27471`
2. A `_dispatch_ready_jobs()` call on every poll tick

The existing `synlynk watch` command stays unchanged — the daemon is a superset, not a replacement.

**Process model:** double-fork (inherited from `WatchDaemon`), two threads in the daemon process:
- **Poll thread** (main loop, inherited): every 3s — check mtimes, regenerate `context.md` on change, call `_reconcile_jobs()`, call `_dispatch_ready_jobs()`
- **HTTP thread** (new): `http.server.HTTPServer.serve_forever()` on `localhost:27471`, started before the poll loop enters

**Files:**
- Pidfile: `.synlynk/daemon.pid` (separate from `watch.pid`)
- Log: `.synlynk/daemon.log`

If `watch.pid` exists when daemon starts, print a warning and continue — both poll `project-docs/` and write `context.md`; last-write-wins is acceptable.

## Section 2: Job Queue Schema

New table `daemon_jobs` in `state.db`:

```sql
CREATE TABLE IF NOT EXISTS daemon_jobs (
    job_id       TEXT PRIMARY KEY,
    agent        TEXT NOT NULL,
    task         TEXT NOT NULL,
    story_id     TEXT,
    status       TEXT NOT NULL DEFAULT 'queued',
    priority     INTEGER NOT NULL DEFAULT 5,
    depends_on   TEXT NOT NULL DEFAULT '[]',
    pid          INTEGER,
    enqueued_at  TEXT NOT NULL,
    started_at   TEXT,
    completed_at TEXT,
    exit_code    INTEGER,
    log_path     TEXT
);
```

**Status values:** `queued` → `running` → `done` | `failed`

**Priority:** integer 1–10, lower = higher priority. Default 5.

**depends_on:** JSON array of `job_id` strings. A job stays `queued` until every job_id in `depends_on` has `status='done'`.

**Dispatch logic (`_dispatch_ready_jobs`):**
1. Count `status='running'` rows → if `>= max_parallel`, return early
2. Select `status='queued'` ordered by `priority ASC, enqueued_at ASC`
3. For each candidate, check all `depends_on` IDs are `done` — skip if not
4. Launch via `subprocess.Popen([agent_cmd, ...], start_new_session=True)` — same pattern used by existing job dispatch in `exec_command()`
5. Update row: `status='running'`, `pid=<pid>`, `started_at=<iso>`

Job stdout/stderr → `.synlynk/jobs/<job_id>.log` (same pattern as existing jobs subsystem).

`_reconcile_jobs()` (inherited) reaps finished subprocesses and updates `status`, `exit_code`, `completed_at`.

## Section 3: HTTP API

Server: `http.server.BaseHTTPRequestHandler` subclass — stdlib only, no third-party deps.

Bound to `127.0.0.1:27471` only. No authentication.

All responses `Content-Type: application/json` unless `Accept: text/plain` requested on `/context`.

| Method | Path | Response |
|--------|------|----------|
| `GET` | `/context` | `context.md` content as `{"content": "..."}` or plain text |
| `GET` | `/status` | `{running, uptime_s, pid, jobs: {queued, running, done, failed}}` |
| `GET` | `/jobs` | Array of all jobs; accepts `?status=` filter |
| `GET` | `/jobs/<id>` | Single job object + `log_tail` (last 100 lines) |
| `POST` | `/dispatch` | Body: `{agent, task, story_id?, priority?, depends_on?}` → `{job_id}` |
| `GET` | `/stories` | Story list from `state.db` |
| `GET` | `/stories/<id>` | Single story with all fields |
| `GET` | `/capability` | Capability ratings grouped by `agent × domain` |
| `GET` | `/sentinel` | Contents of `sentinel.md` parsed as JSON array of alert objects |
| `POST` | `/checkpoint` | Force-regenerates `context.md` immediately → `{regenerated: true}` |

Error responses: `{"error": "<message>"}` with appropriate HTTP status codes (404 for unknown job/story, 400 for bad request body, 500 for internal errors).

## Section 4: Service Installation

`synlynk daemon --install-service` detects platform and writes the appropriate unit, then activates it.

### macOS (launchd)

Writes `~/Library/LaunchAgents/com.synlynk.daemon.plist`:

```xml
<?xml version="1.0" encoding="UTF-8"?>
<!DOCTYPE plist PUBLIC "-//Apple//DTD PLIST 1.0//EN" "http://www.apple.com/DTDs/PropertyList-1.0.dtd">
<plist version="1.0">
<dict>
    <key>Label</key><string>com.synlynk.daemon</string>
    <key>ProgramArguments</key>
    <array>
        <string>/path/to/synlynk</string>
        <string>daemon</string>
        <string>start</string>
    </array>
    <key>RunAtLoad</key><true/>
    <key>KeepAlive</key><false/>
    <key>StandardOutPath</key><string>~/.synlynk/launchd.log</string>
    <key>StandardErrorPath</key><string>~/.synlynk/launchd.log</string>
</dict>
</plist>
```

`KeepAlive=false` because synlynk manages its own daemonization — launchd only needs to launch it once at login.

Then runs: `launchctl load -w ~/Library/LaunchAgents/com.synlynk.daemon.plist`

### Linux (systemd user unit)

Writes `~/.config/systemd/user/synlynk-daemon.service`:

```ini
[Unit]
Description=synlynk daemon
After=default.target

[Service]
Type=forking
ExecStart=/path/to/synlynk daemon start
PIDFile=%h/.synlynk/daemon.pid
Restart=on-failure

[Install]
WantedBy=default.target
```

Then runs: `systemctl --user enable --now synlynk-daemon`

Detection: `shutil.which("systemctl")` returns a path.

### Fallback (crontab)

If neither launchd nor systemd is available:

```bash
(crontab -l 2>/dev/null; echo "@reboot /path/to/synlynk daemon start") | crontab -
```

Same pattern as existing `_install_cron_entry()`.

### Uninstall

`synlynk daemon --uninstall-service` reverses each:
- macOS: `launchctl unload <plist>` + `os.remove(plist_path)`
- Linux: `systemctl --user disable --now synlynk-daemon` + `os.remove(unit_path)`
- crontab: read, filter out the `@reboot synlynk daemon start` line, write back

## Section 5: CLI Surface

```
synlynk daemon start               fork-daemonize, write .synlynk/daemon.pid
synlynk daemon stop                SIGTERM to pid from pidfile
synlynk daemon status              running/stopped + queue depth + HTTP URL
synlynk daemon restart             stop then start
synlynk daemon --install-service   register with launchd/systemd/crontab
synlynk daemon --uninstall-service deregister
```

**`daemon status` output (when running):**
```
  ✦ synlynk daemon running  (pid 12345, up 2h 14m)
    jobs: 0 queued · 1 running · 47 done · 0 failed
    http: http://localhost:27471
```

**`daemon status` output (when stopped):**
```
  ✦ synlynk daemon not running
```

**`daemon stop` when not running:** prints `  ✦ daemon not running` and exits 0 (idempotent).

**Coexistence with `synlynk watch`:** Both commands can run simultaneously. `daemon start` checks for `watch.pid` and prints a warning if found, but does not block.

## Out of Scope (v0.9.3)

- Job cancellation via HTTP (`DELETE /jobs/<id>`) — deferred to v0.9.4
- Webhook/push notifications when jobs complete — deferred
- Multi-project daemon (one daemon per project-dir) — current design; each project runs its own daemon
- Auth/TLS on the HTTP API — localhost-only, no auth needed for v0.9.3
