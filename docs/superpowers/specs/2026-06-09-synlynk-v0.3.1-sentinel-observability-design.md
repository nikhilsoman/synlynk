# synlynk v0.3.1 — Sentinel + Observability Hardening: Design Spec

**Date:** 2026-06-09  
**Status:** Approved  
**Author:** Nikhil Soman + Claude Sonnet 4.6  
**Roadmap ref:** Unified Roadmap §5 — v0.3.1

---

## 1. Goal

Restore two regressions from v1.2.x and add a cohesive set of agent health signals so that `synlynk status` becomes a trustworthy health dashboard — surfacing zombie daemons, looping agents, quota exhaustion, cost burn rate, and context bloat before the user has to notice them manually.

**Required before v0.5.0:** `extract_tokens()` feeds `agent_quotas.used_tokens` in the v0.5.0 state.db schema. This patch must land first.

---

## 2. Constraints

- **Single file** — all logic stays in `bin/synlynk.py`. Zero new dependencies (stdlib only).
- **No background threads** — all detection runs at call time (`exec`, `status`, `sentinel`). No new daemon.
- **TTY preserved** — interactive exec (Claude Code, Gemini CLI) must continue to work. Token scraping uses a non-blocking tee approach for non-interactive mode only; interactive mode is auto-detected.
- **Additive** — no existing command changes its exit code or output format in a way that breaks scripts.

---

## 3. Features

### 3.1 Token scraping + cost pulse (regression fix)

**Problem:** v0.3.0 switched exec to TTY pass-through (`subprocess.Popen` with inherited stdio) to support interactive CLIs. This broke stdout capture, so `extract_tokens()` and `update_costs()` were removed. The v0.5.0 state.db spec assumes `extract_tokens()` exists.

**Design:**

Detect interactive vs non-interactive at exec time:

```python
def _is_interactive(cmd_args: list) -> bool:
    NON_INTERACTIVE_FLAGS = ["--no-tty", "--output-format json", "-p ", "--print"]
    cmd = " ".join(cmd_args)
    return not any(f in cmd for f in NON_INTERACTIVE_FLAGS)
```

- **Interactive mode** (default): TTY pass-through unchanged. After exec completes, attempt to parse the AI CLI's own log file if one exists (Claude writes `~/.claude/logs/`, Gemini writes to stderr summary lines). If no log: skip token scraping, print `  ⚡ Token count unavailable (interactive mode — use --print flag for cost tracking)` once.
- **Non-interactive mode** (detected via flags): capture stdout via `subprocess.Popen(stdout=PIPE, stderr=STDOUT)`, tee to terminal in real time using a reader thread, scrape buffer after completion.

**`extract_tokens(output_text: str) -> tuple[int, int]`** — restored from v1.2.x with extended patterns:

| Pattern | CLI |
|---|---|
| `Input tokens: N` / `Output tokens: N` | claude |
| `"input_tokens": N` | claude JSON mode |
| `Tokens used: N input, N output` | gemini |
| `prompt_tokens: N` / `completion_tokens: N` | openai-compat |
| `Total tokens: N` (split 80/20 input/output) | fallback |

**`update_costs(command, in_tokens, out_tokens, duration)`** — restored. Appends to `project-docs/costs.md`. Rates: `$0.003/1K input + $0.015/1K output` (same as v1.2.x; update in function directly if rates change).

**Cost pulse** — printed after every exec where `in_tokens > 0`:
```
  ⚡ Tokens: 4,821 in / 312 out  |  est. $0.0191  |  duration: 47.3s
```
If tokens unavailable: silent (no noise for zero-token runs).

### 3.2 Zombie daemon detection

**Problem:** `synlynk watch` forks a daemon and writes `.synlynk/watch.pid`. If the process dies (OOM, SIGKILL, crash), the pidfile lingers. `_is_running()` does a `os.kill(pid, 0)` check, so it correctly detects stale pids — but does not write a sentinel alert or surface the condition clearly.

**Design:**

Extend `WatchDaemon._is_running()` to return a tri-state:

```python
def _health(self) -> str:  # "running" | "stopped" | "zombie"
```

Add `check_daemon_health()` function called from `exec_command` and `cmd_status`:

```python
def check_daemon_health() -> None:
    daemon = WatchDaemon()
    if daemon._health() == "zombie":
        _write_sentinel_alert("CRITICAL", "ZOMBIE_DAEMON",
            "Watch daemon pidfile exists but process is dead. "
            "Run: synlynk watch stop && synlynk watch start")
```

### 3.3 Stall detection

**Problem:** An exec that hangs indefinitely (waiting for input it will never get, network timeout, deadlock) holds the terminal hostage. No mechanism currently flags long-running execs.

**Design:**

`exec_command` records start time in telemetry. On the *next* call to `cmd_status` or `exec_command`, check the most recent open exec event:

```python

    # If age > config["exec_timeout_minutes"] (default: 30), write WARN alert
```

Config addition to `.synlynk/config.json`:
```json
{ "exec_timeout_minutes": 30 }
```

`init()` writes this default. Existing installs: `load_config()` uses `.get("exec_timeout_minutes", 30)`.

Note: stall detection is retrospective (detected on next invocation), not real-time. This is intentional — no background thread required.

### 3.4 Quota-exhausted exit detection

**Problem:** When an AI CLI hits a rate limit or billing cap it exits non-zero, but synlynk logs this as a generic failure. The flatline detector fires after 3 repeats — but the root cause (quota) is never identified.

**Design:**

After exec completes with non-zero exit, scan captured output (or stderr if available) for known quota phrases:

```python
QUOTA_PATTERNS = {
    "claude":  ["rate limit", "usage limit", "quota exceeded", "billing", "overloaded"],
    "gemini":  ["quota exceeded", "resource exhausted", "RESOURCE_EXHAUSTED"],
    "codex":   ["rate limit exceeded", "insufficient_quota"],
    "_any":    ["rate limit", "quota", "billing limit", "too many requests"],
}
```

Match is case-insensitive. On match: write CRITICAL sentinel alert:
```
CRITICAL [2026-06-09 14:23] QUOTA_EXHAUSTED: claude — matched "rate limit exceeded". 
Consider: wait for reset, switch agent CLI, or increase plan limits.
```

In interactive mode where stdout isn't captured: check exit code only. Exit code 429 from claude → CRITICAL. Exit code 1 with known quota-duration heuristic (session < 5s) → WARN.

### 3.5 Success loop detection

**Problem:** Current flatline detection catches repeated *failures*. Repeated *successes* of the same command in rapid succession can indicate an automated loop burning tokens with no progress (e.g., a misconfigured script calling `synlynk exec claude` in a tight loop).

**Design:**

In `check_flatline()` (rename to `check_sentinel_patterns()` to cover all patterns), add a second check after the failure check:

```python
# Success loop: last 5 execs same command, all exit 0, within 10 minutes
last_five = data[-5:]
if (len(last_five) == 5 and
    all(e.get("exit_code", 1) == 0 for e in last_five) and
    all(e.get("command") == last_five[0].get("command") for e in last_five)):
    window_minutes = (time.time() - last_five[0].get("_ts", 0)) / 60
    if window_minutes < 10:
        _write_sentinel_alert("WARN", "SUCCESS_LOOP",
            f"Same command succeeded 5x in {window_minutes:.1f}min. "
            "Possible automated loop burning tokens.")
```

Telemetry events gain `_ts` (Unix timestamp float) for duration calculations. Existing events without `_ts` are skipped gracefully.

### 3.7 Context bloat warning

**Problem:** `context.md` grows unbounded as project-docs/ grows. Large context = higher per-exec token cost + reduced agent quality.

**Design:**

In `generate_context()`, after writing `.synlynk/context.md`, check size:

```python
size_kb = os.path.getsize(".synlynk/context.md") / 1024
if size_kb > 32:
    print(f"  ⚠ Context is large ({size_kb:.0f} KB) — consider archiving "
          "completed todos and old devlog entries to reduce token cost.")
```

Thresholds: warn at 32 KB, repeat warn at 64 KB with stronger language. No blocking — informational only.

---

### 3.8 Sentinel severity + ack

**Problem:** `sentinel.md` accumulates alerts forever. No way to dismiss resolved alerts. No severity distinction — flatline and zombie look identical.

**Design:**

**Severity levels:** `INFO` | `WARN` | `CRITICAL`

Alert format (replacing current free-form line):
```
- [CRITICAL] [2026-06-09 14:23] ZOMBIE_DAEMON: Watch daemon pidfile exists but process is dead. Run: synlynk watch stop && synlynk watch start
- [WARN]     [2026-06-09 14:10] SUCCESS_LOOP: Same command succeeded 5x in 3.2min.
```

**`_write_sentinel_alert(severity, code, message)`** — shared function replacing ad-hoc writes:
```python
def _write_sentinel_alert(severity: str, code: str, message: str) -> None:
```

**`synlynk sentinel list`** — prints active alerts with age since written.

**`synlynk sentinel clear [--severity CRITICAL|WARN|INFO] [--code CODE]`** — removes matching alerts from sentinel.md. No args = clear all. Prints count cleared.

**Backward compatibility:** existing free-form lines in sentinel.md are preserved as-is (treated as INFO, no code). `sentinel clear` skips lines it can't parse.

### 3.9 Pre-exec gate

**Problem:** If a CRITICAL sentinel alert is active (zombie daemon, quota exhausted), launching another exec compounds the problem silently.

**Design:**

In `exec_command()`, after `check_budgets()`:

```python
def _check_pre_exec_gate(force: bool = False) -> bool:
    # Returns True = proceed, False = abort
    sentinel_file = ".synlynk/sentinel.md"
    criticals = _read_sentinel_alerts(severity="CRITICAL")
    warns = _read_sentinel_alerts(severity="WARN")
    if warns:
        for w in warns: print(f"  ⚠ {w}")
    if criticals:
        for c in criticals: print(f"  🚨 {c}")
        if not force:
            print("  Exec blocked by CRITICAL alert. Fix the issue or run with --force.")
            return False
    return True
```

`exec` subparser gains `--force` flag. WARN alerts print but never block.

## 4. Sentinel Alert Codes Reference

| Code | Severity | Trigger |
|---|---|---|
| `FLATLINE` | CRITICAL | 3 consecutive failures of same command |
| `ZOMBIE_DAEMON` | CRITICAL | Pidfile exists, process dead |
| `QUOTA_EXHAUSTED` | CRITICAL | Known quota/rate-limit phrase in CLI output |
| `SUCCESS_LOOP` | WARN | 5 identical successful execs within 10 min |
| `STALL` | WARN | Exec running longer than `exec_timeout_minutes` |
| `BUDGET_WARNING` | WARN | Spend > 80% of `limit_usd` (existing, now uses `_write_sentinel_alert`) |
| `CONTEXT_BLOAT` | INFO | `context.md` > 32 KB |
| `COSTS_STALE` | INFO | costs.md not updated this session (existing, now uses `_write_sentinel_alert`) |
