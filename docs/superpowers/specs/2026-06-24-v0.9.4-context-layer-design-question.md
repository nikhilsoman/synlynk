# v0.9.4 Design Question: When Does Context Move Off Markdown?

**Date:** 2026-06-24
**Status:** Pre-brainstorm framing — not yet approved for implementation
**Blocking:** v0.9.4 Workgroup Relay design cannot be finalised without answering this

---

## Why This Question Surfaces Now

v0.9.3 shipped a daemon with an HTTP API on `localhost:27471`. The `/context` endpoint
currently serves the contents of `.synlynk/context.md` — a flat markdown file that
`generate_context()` assembles by reading several `project-docs/` markdown files on
every poll tick.

v0.9.4 adds a network relay: agents on different machines will pull context over WSS.
Sending a markdown blob over the wire works, but it's the worst version of what the relay
could carry. Before designing the relay wire protocol, we need to decide what *unit of
context* the relay transports — and that decision determines how much of the current
markdown stack gets replaced by SQLite queries.

---

## Current State: What Lives Where

### In `project-docs/` (markdown files, git-tracked)

| File | Contents | Read by `generate_context()`? |
|------|----------|-------------------------------|
| `todo.md` | Active + deferred tasks (`- [ ]`, `- [-]`) | ✅ Yes — active/deferred tasks section |
| `roadmap.md` | Version arc, status per release | ✅ Yes — "In Progress" rows only |
| `memory.md` | Curated decisions + context | ✅ Yes — full content |
| `devlogs/<user>.md` | Per-user session entries | ✅ Yes — last N entries per user |
| `costs.md` | Per-session token/cost rows | ❌ No |
| `source-map.md` | Source symbol index (generated) | ❌ No (read from SQLite scan cache) |
| `sentinel.md` | Active alerts | ✅ Yes — active `- [` lines |

### In `state.db` (SQLite WAL, machine-queryable)

| Table | Contents | In context today? |
|-------|----------|-------------------|
| `stories` | Work units with status, agent assignment, token budget | ❌ No |
| `capability_ratings` | Agent × domain × quality scores, Ed25519-signed | ❌ No |
| `daemon_jobs` | Background job queue with priority + deps | ❌ No |
| `source_symbols` | Language-agnostic symbol index (v0.7) | Partially — via `_check_scan_cache()` |
| `autopilot_runs` | Support engineer run history | ❌ No |

### The gap

The most agent-useful data — **active stories, assigned agent, token budget, capability
scores** — lives in SQLite and is entirely absent from `context.md`. Meanwhile the least
structured data — `memory.md`, raw devlog entries — flows into every context verbatim.

---

## The Forcing Function: Relay Wire Protocol

The v0.9.4 relay has to decide what to synchronise across machines. Three options:

**Option A — Relay the markdown blob**
Relay transports `.synlynk/context.md` as a string. Simple, works today.
Problem: markdown is already a lossy aggregation. The relay can't do per-agent
context scoping, story filtering, or incremental updates — it just blasts the whole
blob to every subscriber on every change.

**Option B — Relay structured facts, assemble markdown at edge**
Each machine's relay node publishes a structured event stream: story updates, task
updates, capability deltas, sentinel alerts. Receiving nodes update their local SQLite
and regenerate `context.md` locally. `generate_context()` becomes a read from SQLite,
not a read from files.

This is the "SQLite as local JetStream" model sketched in the v1 architecture spec:
> "Layer 1 (Solo): synlynk daemon IS the leaf node, minus the network hop (SQLite is
> the local JetStream)"

Problem: requires migrating `generate_context()` off file reads.

**Option C — Dual-read transition**
Relay transports structured events (Option B), but `generate_context()` continues
reading from both files and SQLite during a transition period. Files stay as the
human-editable source; SQLite is populated from both writes and file-watch imports.
Relay consumes SQLite only.

## The Core Design Decisions

### Decision 1: What is the unit of context?

Today: a monolithic markdown string assembled from 5 file reads.

Options:
- **1A. Keep it monolithic** — stay with `context.md`; relay blasts the whole thing
- **1B. Structured slices** — context is a set of typed sections (tasks, stories, memory,
  alerts, source arch) that can be composed per-recipient and updated incrementally
- **1C. Queryable** — no `context.md` at all; agents query `/context?scope=tasks` or
  `/context?story=story-abc123` and get only what they need

### Decision 2: When do `project-docs/` markdown files become read-only outputs?

Currently they are the *source of truth* that generates SQLite state.
The question is whether SQLite becomes the source of truth and the files become a
human-readable view.

Options:
- **2A. Files stay primary** — SQLite is always derived from files; relay blasts file-derived context
- **2B. Files become a view** — agents write to SQLite via API; files are regenerated on
  checkpoint for human review and git history
- **2C. Dual-write indefinitely** — humans edit files; agents write to SQLite; sync runs
  on each exec/checkpoint

### Decision 3: Where does `generate_context()` read from at v0.9.4?

Options:
- **3A. No change** — still reads files; relay carries the markdown output
- **3B. Hybrid** — reads tasks/stories from SQLite; reads memory/devlogs from files
  (those don't have a DB representation yet)
- **3C. Full SQLite** — everything in SQLite; files are generated outputs only

---

## What Needs to be True for the Relay to Work

Minimum viable relay (v0.9.4) needs:

1. **A wire-safe context unit** — something the relay can serialise, diff, and deliver to
   a remote peer without blasting the entire markdown on every change
2. **Per-agent scoping** — a remote agent should receive context relevant to its assignment,
   not the full team context
3. **Incremental updates** — sending a full `context.md` snapshot on every story update
   would be noisy; the relay should carry deltas

None of these three properties are achievable with the current markdown blob approach.
The relay design *requires* at minimum Decision 1B (structured slices).

---

## Recommended Starting Position for Brainstorm

The path of least regret:

1. **v0.9.4: Hybrid read (Decision 3B)** — `generate_context()` reads tasks + stories
   from SQLite (they're already there); continues reading `memory.md` and devlogs from
   files (no DB schema exists for them yet). This unlocks per-agent scoping and structured
   relay events without requiring a full migration.

2. **v0.9.4 relay protocol: structured events (Decision 1B)** — relay publishes typed
   events (`story_updated`, `task_updated`, `alert_raised`) not markdown diffs. Receiving
   nodes merge into local SQLite and regenerate `context.md` as a view.

3. **Files stay primary for human content (Decision 2C near-term, 2B long-term)** —
   `memory.md` and devlogs remain human-editable files through v1.0. After v1.0, a
   `decisions` table in SQLite can absorb `memory.md` and a `devlog_entries` table can
   absorb per-user devlog entries — at which point files become generated outputs.

4. **No `context.md` as the relay unit** — the relay never sends `context.md`. It sends
   events; each node's daemon assembles `context.md` locally for backwards compatibility
   with agents that still call `GET /context` expecting markdown.

---

## Open Questions for Brainstorm

1. Do `memory.md` and devlogs need a SQLite representation at v0.9.4, or can hybrid-read
   defer that to v1.0?

2. What is the schema for relay events? Typed rows per table, or a generic
   `(type, payload, timestamp, origin_node)` event log?

3. Does the relay need to handle conflict resolution for `memory.md` edits (two humans
   editing on different machines simultaneously)? Or is that a v1.0 problem?

4. Should `GET /context` on the daemon continue to return markdown for backwards
   compatibility, with a new `GET /context.json` endpoint returning structured slices?

5. What's the migration path for existing projects — do their `todo.md` tasks get
   imported into a `tasks` table on daemon start, or is that a separate `synlynk migrate`
   command?
