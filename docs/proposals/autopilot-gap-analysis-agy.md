# Synlynk Autopilot Initiative: Gap Analysis

This document analyzes the gap between the proposed Autopilot Initiative (strategic agentic automation framework) and the current state of synlynk as of v0.7.0 (Static Scan Quality and Task Status Model).

---

## Ownership Model

### 1. Codebase / Blog Post Alignment
**Status:** **Unanswered.** 
The current codebase and blog posts do not define how agent ownership maps to `state.db`. While v0.4.2 (Task Status Model) provides the vocabulary (`[ ]`, `[x]`, `[-]`, `[~]`, `[>]`) for tasks inside `project-docs/todo.md` and v0.7.0 stores source map metadata, there is no system implementation or database schema representation details for agent-specific ownership of tasks, stories, or epics in `state.db`.

### 2. Specific Unblocking Decisions
- **Option A (In-Band / DB Integration):** Extend the `state.db` schema (e.g., a `tasks` or `epics` table) with an `assigned_agent_id` column. Under this approach, agents are first-class assignees in the database PM hierarchy.
- **Option B (Out-of-Band / Flat-File Integration):** Agents operate exclusively on file-based context (`project-docs/todo.md`) by parsing task descriptions, matching metadata tags (e.g., `- [ ] @support Run regression scan`), and reporting status inline.
- **Recommended Decision:** Implement **Option A** for long-term telemetry and capability scoring (v0.6.1 Capability Ledger), but use **Option B** as a lightweight bridge for rapid development.

### 3. Blocked Agents
- **PM Agent (Primary):** Needs to know the bounds of its authority to create, assign, and organize epics in `state.db`.
- **Support Engineer & Marketing Intern (Secondary):** Cannot query `state.db` to pull assigned work packages or report finished tasks back to a centralized project hierarchy.

## Human-in-the-Loop Thresholds

### 1. Codebase / Blog Post Alignment
**Status:** **Unanswered.**
The strategic roadmap in Document 1 targets a "Human approval bridge (email)" for v0.7.0, and `synlynk exec` has budget + health gates as of v0.3.1. However, the codebase does not yet define the specific rules or thresholds governing when an agent can execute an action autonomously versus when it must escalate to a human.

### 2. Specific Unblocking Decisions
- Define a structured policy file (e.g., `.synlynk/entitlements.yaml` or `.agents/rules.yaml`) specifying permission levels for standard side-effects:
  - `git.commit`: Allow auto-dispatch? (Yes/No/Sandbox)
  - `git.push` / `github.merge_pr`: Require manual human gate? (Yes/No)
  - `http.post` (e.g., publish blog): Must require double-signature or human email approval bridge approval? (Yes/No)
- Specify the API integration for the approval bridge (e.g., standardizing an email callback hook or webhook format).

### 3. Blocked Agents
- **Marketing Intern (Critical):** Cannot safely publish blog posts to dev.to, Medium, or X without a hard approval gate to prevent hallucinated/incorrect drafts from going public.
- **Support Engineer (High):** Cannot safely file Github issues or merge auto-fixes without hitting a threshold policy.
- **PM Agent (High):** Cannot autonomously modify roadmap goals or commit resources without human approval.

## Agent Personas

### 1. Codebase / Blog Post Alignment
**Status:** **Unanswered.**
The strategic roadmap places "Agent identity + dispatch" in v0.5.0, but the exact mechanism of identity is not detailed. There is no concept of agent personas, cryptographic signatures, or role definitions in the current v0.7.0 implementation.

### 2. Specific Unblocking Decisions
- Establish whether agents have distinct cryptographic identities (e.g., Ed25519 public/private keypairs stored in `.synlynk/identities/<agent-name>/`) used to sign git commits, PR approvals, and telemetry payloads.
- Or, establish if agents run under the host user's local credentials (e.g., Nikhil's git profile) with their persona configured purely as system prompts (capability profiles) in `.agents/`.
- **Recommended Decision:** Use Ed25519 keypairs per agent. This enables auditability in git history (verifying which commits were generated by which agent) and prevents spoofing in multi-agent environments.

### 3. Blocked Agents
- **Marketing Intern:** Blocked because it needs platform-specific API keys and distinct publisher profiles to avoid posting under personal human developer profiles.
- **Support Engineer:** Blocked from pushing auto-fixes or creating regression commits that can be verified and isolated from human-committed code.
- **PM Agent:** Blocked from signing off on release candidates or milestones in the git/GitHub release chain.

## Feedback Loops

### 1. Codebase / Blog Post Alignment
**Status:** **Unanswered.**
The current codebase lacks any agent-to-agent communication protocol or telemetry loop. The v0.4.2 blog post mentions the broader goal of moving synlynk to "operate itself — dispatching Claude, AGY, and Codex on real tasks" but does not define collaboration or feedback mechanisms.

### 2. Specific Unblocking Decisions
- Establish a message-passing primitive or event bus (e.g., an `agent_messages` table in `state.db` or a shared inbox folder `project-docs/agent-inbox/` with flat-file JSON payloads).
- Standardize the metric reporting format so the Marketing Intern can log post-performance metrics (e.g., views, stars) in a shared database, which the PM Agent reads to direct the roadmap.

### 3. Blocked Agents
- **PM Agent:** Cannot dynamically adjust content strategy or direct the Marketing Intern to write about specific features that are driving the most stars/installs.
- **Marketing Intern:** Runs in a silo, unable to receive dynamic topic recommendations or performance evaluations from the PM Agent.

## Cold Start

### 1. Codebase / Blog Post Alignment
**Status:** **Unanswered.**
The codebase is currently geared toward execution inside an established project (`synlynk exec` reads from existing `project-docs/`). There are no templates or bootstrap logic to provision agents on a clean synlynk installation.

### 2. Specific Unblocking Decisions
- Create a bootstrap initialization command (e.g., `synlynk init-autopilot`) that seeds a new repository with:
  1. Default agent configuration files (`.agents/pm.yaml`, `.agents/marketing.yaml`, `.agents/support.yaml`).
  2. The initial PM hierarchy skeleton in `project-docs/` (`roadmap.md`, `todo.md`, `memory.md`).
  3. Pre-configured verification steps and cron configs.

### 3. Blocked Agents
- **Support Engineer:** Needs a baseline configuration specifying the E2E test commands, target branch, and failure threshold triggers.
- **Marketing Intern & PM Agent:** Cannot bootstrap contextually without default templates, RSS/feed targets, and metric sources.

## Recommended First Agent to Launch

The **Support Engineer Agent** is the recommended first agent to launch.

### Rationale:
1. **Infrastructure Readiness:** The Support Engineer agent only requires the primitives available in v0.3.1 (`synlynk exec` + budget/health gates) and v0.4.0 (Trio Protocol - Architect/Build/Verify pipeline). It does not require complex platform integrations (like MCP servers at v0.8.0) or advanced state-db PM hierarchies (v0.6.0+).
2. **Deterministic Inputs/Outputs:** Unlike the PM Agent (which consumes ambiguous metrics and interacts with human roadmaps) or the Marketing Intern (which writes subjective prose), the Support Engineer operates on a deterministic boundary: running tests, detecting failures (`non-zero exit codes`), and bisecting commits.
3. **Safety and Low Risk:** Filing a GitHub issue or reporting a test failure carries almost zero reputational or operational risk compared to publishing blogs (Marketing Intern) or altering project priorities (PM Agent). It provides the safest sandbox for testing the "Human-in-the-loop" approval gates and "Sentinel alert" mechanisms before executing complex write operations.
