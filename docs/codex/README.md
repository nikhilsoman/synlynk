# Synlynk Native Codex Support

This folder is the planning and operating home for Synlynk's native Codex support across:

- Codex CLI interactive sessions.
- Codex CLI non-interactive `codex exec --json` workflows.
- Codex Desktop App local project sessions.
- Codex usage from third-party IDEs and editor shells.

## Ownership

The OpenAI/Codex platform representative owns:

- Native Codex instruction files, especially `AGENTS.md`.
- Codex readiness diagnostics.
- Codex CLI wrappers and event parsing.
- Codex Desktop App setup guidance and local project actions.
- Reliability contracts for third-party IDE usage.
- Ongoing compatibility tracking as Codex CLI/Desktop behavior evolves.

## Design Principles

- Codex must be first-class, not a generic AI CLI afterthought.
- Synlynk should use Codex-native project instructions (`AGENTS.md`) instead of relying on generic fallback files.
- Codex Desktop support should avoid unverified app configuration formats.
- Codex CLI non-interactive workflows should use structured JSONL output where possible.
- Synlynk should never invent cost values. It may record tokens separately when dollars are unavailable.
- Watcher daemons are optional. Codex instructions should prefer explicit `synlynk status` and `synlynk checkpoint` calls for reliability.

## Current Baseline

Current repo behavior before native Codex work:

- `synlynk exec codex` can launch Codex generically.
- No `AGENTS.md` is generated by `synlynk init`.
- No `synlynk doctor --codex` exists.
- No `synlynk codex` command exists.
- No Codex JSONL parser exists.
- No Codex Desktop App project helper exists.

The immediate goal is to close those gaps without disturbing existing Claude, Gemini, Cursor, or generic CLI behavior.
