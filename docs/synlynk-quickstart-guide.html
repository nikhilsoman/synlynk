What is Synlynk?

Your AI agents,
working as a team.

Synlynk is a single-file Python CLI that wraps AI agents — Claude, AGY (AntiGravity), Codex — and makes them project-aware, cost-tracked, and safe to run in parallel. It injects your project's context before every invocation, guards your budget, detects hallucination loops, lets you dispatch background tasks across multiple agents simultaneously, and migrates your existing docs on first run.

Why Synlynk?

🧠
Context-Aware by Default

🛡️
Hallucination Sentinel

Zero Dependencies
Pure Python 3 stdlib. No pip install, no Docker, no build step. One package. Works in any environment where Python 3 runs.

Getting Started

Install in 60 seconds.

1
Clone & install globally
Adds synlynk to your PATH via ~/.synlynk/bin/. Installs the package to ~/.synlynk/lib/synlynk/. No sudo, no pip, no build.
install
$ git clone https://github.com/nikhilsoman/synlynk
$ cd synlynk && ./install.sh
 
✓ Installed synlynk/ package to ~/.synlynk/lib/
✓ Installed binary to ~/.synlynk/bin/synlynk
✓ PATH updated in ~/.zshrc
✓ synlynk 0.9.1 ready
2
Bootstrap your project — with intelligent doc migration
Run inside any repo. If the repo already has docs (roadmap, memory, READMEs), synlynk finds them and migrates their content instead of writing blank skeletons. Skips files that already exist.
synlynk init (mature repo)
~/my-project $ synlynk init
 
Scanning repository for existing docs...
 
✓ project-docs/roadmap.md (migrated from roadmap.md)
✓ project-docs/memory.md (migrated from rxcc_memory.md)
✓ project-docs/todo.md (generated from git history)

✓ claude → /usr/local/bin/claude
✓ agy → /usr/local/bin/agy
 
→ Ready. Try: synlynk exec claude
⚡
Smart Doc Migration
Synlynk searches for existing docs at the repo root, in project-docs/, and project-prefixed variants (rxcc_memory.md, rxcc-roadmap.md). First match >200 bytes is migrated verbatim — your work, not a blank template. Git history powers a seed roadmap as a last resort.
3
Run your first context-injected session
Synlynk prepends your project context before spawning the agent — the AI already knows your roadmap, decisions, source architecture, and open tasks.

v0.4.0+ · Background Dispatch

Run agents in parallel.
In the background.

Dispatch fires an agent as a background job — non-interactive, captured to a log file, tracked by the job store. Multiple agents can run simultaneously on different tasks. You keep your terminal free.

synlynk dispatch
$ synlynk dispatch claude --task "Write unit tests for the billing module"
✓ Dispatched job-a3f2 (claude) running in background
 
$ synlynk dispatch agy --task "Review billing API for edge cases"
✓ Dispatched job-b7e1 (agy) running in background
 
# Both agents working simultaneously — terminal is yours

Monitor all jobs

jobs
$ synlynk jobs
 
● job-a3f2 claude running 2m14s
● job-b7e1 agy running 1m58s

Tail a job's output

logs
$ synlynk logs job-a3f2
 
Writing test_billing.py...
✓ test_charge_success
✓ test_refund_partial
→ 12/12 tests passing

Scoped Story Dispatch v0.9.0

scoped dispatch

Chapter 04 — Safety Systems
SYNLYNK
05
Sentinel & Budget Guard

Your safety net,
always on.

Synlynk watches every agent invocation for danger signals — spiralling costs, hallucination loops, and quota exhaustion — and surfaces alerts before they become expensive mistakes.

💰 Budget Guard

Set per-session and per-day limits in .synlynk/config.json. Synlynk blocks execution the moment you'd exceed your limit.

config.json
{
"limit_usd": 10.00,
"limit_requests": 100,
"project_docs_dir": "project-docs"
}
budget pulse (after each exec)
── Budget Pulse ──────────────
Cost: $2.34 / $10.00
Requests: 14 / 100
Tokens: 2,840 in · 480 out
─────────────────────────────

🛡️ Sentinel Patterns

Sentinel watches your telemetry log and fires alerts for dangerous patterns automatically.

🚨 FLATLINE
Same command failed 3× in a row. Possible hallucination loop. Manual intervention recommended.
⚠ SUCCESS_LOOP
Same command succeeded 5× in <10 min. Likely an automated loop burning tokens unproductively.
🚨 QUOTA_EXHAUSTED
Output matched a known quota-exhaustion phrase. Switch agent or check your plan limits.
💀 ZOMBIE / ⏳ STALL
Job PID is dead but never marked complete (zombie), or job has been running >30 min without output (stall).

Managing sentinel alerts

sentinel commands
$ synlynk sentinel list
 
[CRITICAL] FLATLINE `synlynk exec claude` failed 3x — 2026-06-22 09:41
[WARN] SUCCESS_LOOP Same command 5x in 4.2 min — 2026-06-22 10:02
 
$ synlynk sentinel clear --code FLATLINE
✓ Cleared FLATLINE alerts
🛡️
Achievement Unlocked
Incident Avoided
Sentinel detected a FLATLINE loop before you burned $40 in repeated failed calls. The budget guard has your back.

Team Mode

Every teammate.
Every agent. In sync.

Team mode adds per-user devlogs, a shared roadmap visible to all agents, and a pull-before-write guard that prevents doc conflicts in concurrent sessions. One command to activate it.

enable team mode
$ synlynk init --mode team --org acme --repo api
 
✓ Team mode enabled — project-docs/.synlynk_config.json updated
✓ project-docs/devlogs/nikhil.md created
✓ CLAUDE.md updated with team session protocol
✓ Pull-before-write guard active

How teams work with Synlynk

📒
Per-User Devlogs
Each team member writes to their own devlogs/username.md. Agents read recent entries from all teammates so every AI session knows what the whole team has been doing.
🔒
Pull-Before-Write
Before writing to any shared doc (roadmap, memory, todo), synlynk auto-pulls from origin. Prevents doc divergence in multi-developer repos. Works with git worktrees.
🏷️
Attribution in Memory
Decisions written to memory.md carry [@username] markers. Every agent in the team sees who made each decision and when — accountability without meetings.
📊
Team Status Dashboard
synlynk status in team mode shows each teammate's last active session, recent decisions from their devlog, and any open sentinel alerts — your daily standup in one command.

Session protocol for teams

1
Start every session with a pull
git pull before synlynk exec — the synlynk guard will remind you if you forget.
2
Write decisions to memory with attribution
Add [@yourname 2026-06-22] to any decision line in memory.md — agents surface these to teammates automatically.
3
Update your devlog before committing
A one-line summary in devlogs/yourname.md is all synlynk needs to brief other agents on your work.
4
Use synlynk status for async standups
synlynk status shows the whole team's last 3 sessions, open todos, and active sentinel alerts. Replace the morning standup with a glance at the terminal.
💡
Onboarding a new teammate
Have them clone the repo and run synlynk init — their devlog is created automatically, memory.md and roadmap.md are already there from git. They're context-loaded from the first exec.
Chapter 06 — Monorepo Onboarding
SYNLYNK
07
Monorepos

One repo.
One state database.

Synlynk is monorepo-native. All worktrees for the same git root share a single state database. The scanner excludes node_modules, dist, build, vendor, .next, target, and __pycache__ automatically. All you need to decide is where to put the docs.

Step 1 — Decide where your docs live

Option A: docs in project-docs/ (default)

Docs stay in a dedicated project-docs/ directory at the monorepo root. Best for repos that didn't have docs before synlynk.

default
~/monorepo $ synlynk init
✓ project-docs/roadmap.md
✓ project-docs/memory.md

Option B: docs at repo root (--docs-dir .)

If the monorepo already has roadmap.md, memory.md etc. at the root, tell synlynk where to find them.

root docs
~/monorepo $ synlynk init --docs-dir .
✓ roadmap.md (migrated from roadmap.md)
✓ memory.md (migrated from memory.md)

Step 2 — Verify the state DB path

The state DB lives at ~/.synlynk/projects/<8-char-hash-of-git-root>/state.db. All worktrees for the same monorepo share this DB — capability scores, jobs, and stories are unified across your entire team working on the same repo.

verify state db
$ synlynk status
 
State DB: ~/.synlynk/projects/a1b2c3d4/state.db
Git root: /Users/nikhil/monorepo
Docs dir: project-docs/
Worktrees sharing this DB: 3

Step 3 — Run a source scan

For large monorepos, run an explicit scan so synlynk indexes the architecture before any agent dispatch. The scanner respects all standard build-output exclusions.

synlynk scan
$ synlynk scan --deep
 
Scanning 3,420 files (skipping node_modules, dist, build, .next, vendor...)
✓ Found 847 symbols across 9 languages
✓ Wrote project-docs/source-map.md
✓ Cache stored at HEAD sha a1b2c3d — zero cost on next exec
💡
Monorepo checklist
1. One synlynk init at the git root — not per-package.
2. Use --docs-dir . if docs already exist at root; default otherwise.
3. Run synlynk scan --deep once after init to build the architecture cache.
4. All worktrees share the same state DB — capability scores apply repo-wide.
Chapter 07 — Multi-Repo Onboarding
SYNLYNK
08
Multi-Repo Teams

Multiple repos.
One workflow.

With multi-repo setups, each repository gets its own state database keyed to its git root. Synlynk agents stay scoped to the repo they're running in — no cross-contamination. The pattern is simple: run synlynk init once in each repo, independently.

Each repo is fully independent

init each repo independently
# Repo 1 — API service
~/dev/api $ synlynk init --mode team --org acme --repo api
✓ State DB: ~/.synlynk/projects/f1a2b3c4/state.db
 
# Repo 2 — Frontend app
~/dev/app $ synlynk init --mode team --org acme --repo app
✓ State DB: ~/.synlynk/projects/9e8d7c6b/state.db
 
# Each repo has its own context, budget, and sentinel

Linking repos to the same GitHub Project

Use --project-id to fill the same GitHub Projects v2 node ID into agent instruction files across all repos. This lets you track stories and PRs from different repos in one unified board while each agent file is repo-specific.

linked project boards
~/dev/api $ synlynk init --org acme --repo api --project-id PJ_kwDO1234
✓ CLAUDE.md · GEMINI.md · AGENTS.md — project ID baked in
 
~/dev/app $ synlynk init --org acme --repo app --project-id PJ_kwDO1234
✓ CLAUDE.md · GEMINI.md · AGENTS.md — same project board

Multi-repo workflow tips

📁
Separate Costs per Repo
Each repo has its own project-docs/costs.md and budget limits. You can set a tighter limit on the frontend repo and a generous limit on the core API.
🔍
Scoped Source Scans
Each synlynk scan is scoped to that repo's git root. Agents dispatched in the API repo only see API source files — no frontend noise.
🧠
Shared Standards
Keep a shared AI_INSTRUCTIONS.md template in a central repo. Copy or reference it in each repo's synlynk init run so every agent gets the same standards.
🔀
Cross-Repo at v0.10
Workspace federation — where a single synlynk session can dispatch across multiple repos — is planned for v0.10.0. Until then, dispatch per-repo from separate shells.
⚠️
One init per git root
Never run synlynk init inside a git submodule — it will create a separate state DB keyed to the submodule root, which is not the same as the parent repo's DB. Always run from the repo's own root directory.

Source Scan & Docs_Dir

Source-aware agents.
Docs wherever you need.

🔍 Synlynk Scan v0.7.0

The static scanner indexes your codebase and injects a ## Source Architecture section into every agent dispatch. The cache is keyed by git HEAD SHA — zero overhead on a cache hit. Supports Python, JS, TS, Go, Rust, Ruby, Java, Kotlin, and Shell.

scan commands
# Quick scan — injects ## Source Architecture into context
$ synlynk scan
✓ Scanned 284 files · 1,024 symbols · cache hot (HEAD: a1b2c3d)
 
# Deep scan — full symbol extraction, writes source-map.md
$ synlynk scan --deep
✓ Wrote 284 entries to source_symbols (state.db)
✓ Wrote project-docs/source-map.md (1,024 lines)
 
# Cache status
$ synlynk scan --status
Cache age: 4 min · HEAD sha: a1b2c3d · 284 files · 1,024 symbols

📁 Docs_Dir Customization v0.9.1

By default synlynk creates and reads from project-docs/. Use --docs-dir at init time to point synlynk at any directory — the root (.), a custom path (docs/), or a deeply nested location. The setting is persisted to .synlynk/config.json and respected by all subsequent commands.

set docs dir at init
# Docs at repo root
$ synlynk init --docs-dir .
 
# Docs in a custom folder
$ synlynk init --docs-dir docs/agent
.synlynk/config.json
{
"project_docs_dir": "docs/agent",
"limit_usd": 10.00,
"limit_requests": 100
}
⚡
What synlynk auto-discovers during init
Before writing any file, synlynk searches for existing docs at: 1) the target dir itself, 2) repo root, 3) project-docs/, 4) project-prefixed names like rxcc_memory.md or rxcc-roadmap.md. The first file over 200 bytes wins and is migrated verbatim. This means synlynk respects your existing docs regardless of how they're named or where they live.

Scan exclusions — what is always skipped

always excluded from scan
node_modules/ · dist/ · build/ · .next/ · out/
vendor/ · target/ · __pycache__/ · .venv/ · venv/
.git/ · .synlynk/ · *.min.js · *.bundle.js
 
# Result: your code; not build output

Setup & Context

CommandWhat it does

Dispatch & Jobs

CommandWhat it does

synlynk logs <job-id>Tail the output log of a specific job. Use --tail N to control lines shown.
