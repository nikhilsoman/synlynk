What is Synlynk?

Your AI agents,
working as a team.

Synlynk is a single-file Python CLI that wraps AI agents — Claude, Gemini, Codex — and makes them project-aware, cost-tracked, and safe to run in parallel. It injects your project's context before every invocation, guards your budget, detects hallucination loops, and lets you dispatch background tasks across multiple agents simultaneously.

Why Synlynk?

🧠
Context-Aware by Default
Every agent invocation automatically receives your roadmap, memory notes, and todos — no more copy-pasting context into every session.
💰
Zero Surprise Bills
Budget guard blocks execution the moment you hit your daily or session limit. Real-time cost tracking in every exec.
🛡️
Hallucination Sentinel
Automatically detects flatline loops (3 same-command failures), success loops (5 identical runs), and quota exhaustion. Alerts before damage.
⚡
Parallel Dispatch
Launch Claude, Gemini, and Codex in parallel on different tasks. Monitor all jobs, tail logs, and compare results — all from one CLI.
👥
Team-Ready from Day One
Single or team mode. Per-user devlogs with git attribution. Shared roadmap and memory visible to every agent in the repo.
🔌
Zero Dependencies
Pure Python 3 stdlib. No pip install, no Docker, no build step. One file. Works in any environment where Python 3 runs.

Getting Started

Install in 60 seconds.

1
Clone & install globally
Adds synlynk to your PATH via ~/.synlynk/bin/. No sudo, no pip, no build.
install
$ git clone https://github.com/nikhilsoman/synlynk
$ cd synlynk && ./install.sh

✓ PATH updated in ~/.zshrc
✓ synlynk 0.4.0 ready
2
Bootstrap your project
Run inside any repo. Creates project-docs/, writes CLAUDE.md, GEMINI.md, AI_INSTRUCTIONS.md, and .synlynk/config.json. Skips files that already exist.
synlynk init

Chapter 03 — Multi-Agent Dispatch

v0.4.0 Feature · Hybrid Workgroup Bootstrap

Run agents in parallel.
In the background.

Dispatch fires an agent as a background job — non-interactive, captured to a log file, tracked by the job store. Multiple agents can run simultaneously on different tasks. You keep your terminal free.

synlynk dispatch
$ synlynk dispatch claude --task "Write unit tests for the billing module"
✓ Dispatched job-a3f2 (claude) running in background
 
$ synlynk dispatch gemini --task "Review billing API for edge cases"
✓ Dispatched job-b7e1 (gemini) running in background
 
# Both agents working simultaneously — terminal is yours

Monitor all jobs

jobs
$ synlynk jobs
 
● job-a3f2 claude running 2m14s
● job-b7e1 gemini running 1m58s

Tail a job's output

logs
$ synlynk logs job-a3f2
 
Writing test_billing.py...
✓ test_charge_success
✓ test_refund_partial
→ 12/12 tests passing

The Trio Pipeline v0.4.0

Run a three-agent pipeline: Architect designs, Builder implements, Verifier reviews — all in sequence with full context hand-off between each.

synlynk run --trio

Chapter 04 — Safety Systems
SYNLYNK
05
Sentinel & Budget Guard

Your safety net,
always on.

Synlynk watches every agent invocation for danger signals — spiralling costs, hallucination loops, and quota exhaustion — and surfaces alerts before they become expensive mistakes.

💰 Budget Guard

Set per-session and per-day limits in .synlynk/config.json. Synlynk blocks execution the moment you'd exceed your limit.

config.json
{
"limit_usd": 10.00,
"limit_requests": 100
}
budget pulse (after each exec)
── Budget Pulse ──────────────
Cost: $2.34 / $10.00
Requests: 14 / 100
Tokens: 2,840 in · 480 out
─────────────────────────────

🛡️ Sentinel Patterns

Sentinel watches your telemetry log and fires alerts for dangerous patterns automatically.

🚨 FLATLINE
Same command failed 3× in a row. Possible hallucination loop. Manual intervention recommended.
⚠ SUCCESS_LOOP
Same command succeeded 5× in <10 min. Likely an automated loop burning tokens unproductively.
🚨 QUOTA_EXHAUSTED
Output matched a known quota-exhaustion phrase. Switch agent or check your plan limits.
💀 ZOMBIE / ⏳ STALL
Job PID is dead but never marked complete (zombie), or job has been running >30 min without output (stall).

Managing sentinel alerts

sentinel commands
$ synlynk sentinel list
 
[CRITICAL] FLATLINE `synlynk exec claude` failed 3x — 2026-06-14 09:41
[WARN] SUCCESS_LOOP Same command 5x in 4.2 min — 2026-06-14 10:02
 
$ synlynk sentinel clear --code FLATLINE
✓ Cleared FLATLINE alerts
🛡️
Achievement Unlocked
Incident Avoided
Sentinel detected a FLATLINE loop before you burned $40 in repeated failed calls. The budget guard has your back.
Chapter 05 — Command Reference
SYNLYNK
06
All Commands

Everything at a glance.

Setup & Context

CommandWhat it does
synlynk init Bootstrap project-docs/, agent instruction files, and .synlynk/config.json in any repo. Safe to re-run — skips existing files.
synlynk exec <agent> Run an agent interactively with full context injection (memory + roadmap + todos prepended). Captures token counts and cost on exit.
synlynk checkpoint Force a context snapshot — regenerates .synlynk/context.md from current project-docs state.
synlynk status Dashboard showing current project state, recent costs, active sentinel alerts, and running jobs.
synlynk upgrade Check for and apply updates from the remote repository.

Dispatch & Jobs v0.4.0

CommandWhat it does
synlynk dispatch <agent> Fire an agent task in the background. Captures stdout to a log file. Returns a job ID immediately. Accepts --task "..." and --story <id>.
synlynk jobs List all tracked jobs with status (running / completed / failed), agent, duration. Use --all to include completed jobs.
synlynk logs <job-id> Tail the output log of a specific job. Use --tail N to control how many lines to show.
synlynk shell <agent> Open an interactive shell pre-loaded with the agent's context. Useful for debugging or ad-hoc queries.
synlynk run --trio Run a three-phase Architect → Builder → Verifier pipeline. Sequentially dispatches three jobs with context hand-off between each phase.

Sentinel & Safety

CommandWhat it does
synlynk sentinel list Show all active alerts with severity (CRITICAL / WARN), code, message, and timestamp.
synlynk sentinel clear Clear alerts. Use --severity CRITICAL or --code FLATLINE to target specific alerts.

Project Docs Structure

directory layout

├── roadmap.md ← version plan
├── memory.md ← decisions
├── todo.md ← open tasks
├── costs.md ← cost ledger
└── devlogs/
└── nikhil.md ← per-user log
 
.synlynk/
├── config.json ← budget limits
├── context.md ← auto-generated
├── telemetry.json ← rolling log
└── logs/ ← job outputs
💡
What agents read
Every synlynk exec prepends memory.md + roadmap.md + todo.md into a single context.md snapshot before handing off to the agent. Your agents always know where the project stands.
⚠️
Keep project-docs current
Synlynk's context is only as good as your memory.md and roadmap.md. Update them after significant decisions — agents will pick up the changes on the next exec automatically.
