Design — Section 1: Workspace Structure

One workspace per product · One state.db · Repos are a dimension, not a boundary

File layout — multi-repo product

Machine-level (~/.synlynk/)
~/.synlynk/
  identity.key          ← Ed25519 private key
  identity.json         ← machine_uuid · public_key
                              git_user · git_email
  workspaces/
    my-product/
      state.db           ← ALL workspace state
    another-product/
      state.db
Per-repo (.synlynk/) — lightweight
~/dev/my-product/
  frontend/              ← git repo
    .synlynk/
      config.json       ← { workspace_id,
                              repo_id: "frontend" }
      context.md       ← generated snapshot
  backend/               ← git repo
    .synlynk/
      config.json       ← { workspace_id,
                              repo_id: "backend" }
      context.md
  infra/
    .synlynk/             ← same pattern

Key invariant change from previous design

Before (project-scoped)

identity.key → .synlynk/identity.key
state.db → ~/.synlynk/projects/<project_id>/
One project = one repo
Cross-repo: not supported
After (machine + workspace scoped)

identity.key → ~/.synlynk/identity.key
state.db → ~/.synlynk/workspaces/<id>/
One workspace = N repos (or 1 — transparent)
Cross-repo Epics: first-class

Solo dev experience — zero friction

synlynk init in a single repo creates a workspace with one member. The user sees nothing different — no workspace commands, no new config to understand. The workspace directory exists at ~/.synlynk/workspaces/<id>/ but is invisible until you add a second repo.
