Design — Section 2: Init + Join Flow

Repo-first, auto-promoted · Solo is zero-friction · Multi-repo is one command

Path A — solo dev, single repo (the common case)

synlynk init in frontend/
$ cd ~/dev/my-product/frontend && synlynk init

1. No ~/.synlynk/identity.key found → generate Ed25519 keypair
   Writes ~/.synlynk/identity.key (chmod 600)
   Writes ~/.synlynk/identity.json { machine_uuid, public_key, git_user }

2. No workspace found → create new workspace
   workspace_id = uuid4()
   workspace_name = git remote slug OR parent dir name → "my-product"
   Creates ~/.synlynk/workspaces/my-product/state.db
   Seeds: repos, arcs, phases, epics, roles, entitlements tables

3. Register this repo
   INSERT INTO repos { id="frontend", name="frontend", path=cwd, git_remote }
   Writes .synlynk/config.json { workspace_id, repo_id: "frontend" }

4. Writes project-docs/ templates (conventions.md, roadmap.md, ...)

✓ Workspace "my-product" created · 1 repo · identity ready
  To add another repo: cd ../backend && synlynk workspace join my-product

Path B — adding a second repo to an existing workspace

Option B1 — explicit join
$ cd ~/dev/my-product/backend
$ synlynk workspace join my-product

Workspace "my-product" found locally
Repo name for this directory? [backend] ↵

INSERT INTO repos { id="backend", ... }
Writes .synlynk/config.json { workspace_id, repo_id: "backend" }

✓ backend joined workspace "my-product"
  Workspace now has 2 repos: frontend · backend
Option B2 — auto-detect on init
$ cd ~/dev/my-product/backend
$ synlynk init

GitHub remote: github.com/nikhil/backend
→ same org as "frontend" in workspace "my-product"

Found related workspace "my-product" (frontend)
Join it? [Y/n] Y

INSERT INTO repos { id="backend", ... }
Writes config.json { workspace_id, repo_id: "backend" }

✓ backend joined workspace "my-product"

Path C — inherited multi-repo (team already has 3 repos, no synlynk)

Adoption flow — bring synlynk to an existing multi-repo product
# Step 1: init in any repo first
$ cd ~/dev/existing-product/api && synlynk init
→ creates workspace "existing-product" · repo "api" registered

# Step 2: join remaining repos (any order)
$ cd ../web && synlynk workspace join existing-product
$ cd ../infra && synlynk workspace join existing-product

# Done. No workspace repo, no config file at parent dir.
# Existing git history, branch structure, CI — all untouched.

✓ Workspace "existing-product" · 3 repos: api · web · infra

Workspace management commands

synlynk workspace list
→ all workspaces on this machine

synlynk workspace join <name-or-id>
→ add current repo to existing workspace

synlynk workspace status
→ all repos, last activity, open epics
synlynk workspace export
→ portable JSON snapshot of state
  (pre-Tokq team sync mechanism)

synlynk workspace import <file>
→ merge snapshot into local state

synlynk workspace rename <new-name>

Pre-Tokq team sync — export/import

Gaurav runs synlynk workspace export → sends JSON to Nikhil → synlynk workspace import gaurav-state.json. Last-write-wins per entity (story, event, memory). Conflicts are surfaced, not silently merged. Full sync activates at Tokq Alpha via NATS — the export/import mechanism is the lightweight bridge until then.
