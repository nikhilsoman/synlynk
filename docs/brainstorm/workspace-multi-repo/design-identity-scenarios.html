Design — Section 3: Identity by Scenario

What identity means in practice across four real situations

Scenario 1 — Solo dev, single machine (the common case)

Nikhil · MacBook · one workspace · one identity
~/.synlynk/identity.key    ← nikhil-macbook · Ed25519
~/.synlynk/identity.json   ← { machine_uuid: "m1", git_user: "nikhilsoman" }

~/.synlynk/workspaces/synlynk/state.db
  dispatch_log: all signed with m1 key
  events: all signed with m1 key

✓ Trivial. No multi-identity complexity. Gap 10 closed.
  Tokq Alpha: registers m1 public key against nikhilsoman account.

Scenario 2 — Solo dev, two machines (laptop + desktop)

MacBook (m1)
~/.synlynk/identity.key  ← m1 key
identity.json: machine_uuid="m1"

workspaces/synlynk/state.db
 dispatch_log rows: sig=m1
 events: sig=m1
Desktop (m2)
~/.synlynk/identity.key  ← m2 key
identity.json: machine_uuid="m2"

workspaces/synlynk/state.db
 dispatch_log rows: sig=m2
 events: sig=m2
Pre-Tokq: Two separate state.dbs. Work done on m1 doesn't automatically appear on m2. Sync via synlynk workspace export → copy file → synlynk workspace import. Imported events carry m1 sig — provably from MacBook, not forged on Desktop.

Tokq Alpha: Both m1 + m2 public keys registered under nikhilsoman Tokq account. NATS sync keeps both state.dbs converged. Attribution: "nikhilsoman via m1" vs "nikhilsoman via m2" — both verified.

Scenario 3 — Simulated team (Nikhil + two alt identities, one machine)

One machine · three git identities · one workspace state.db
Identity is machine-level → only ONE keypair on this machine.
How do three "people" sign events differently?

Answer: identity = machine_uuid + git_user.
The git_user at dispatch time is recorded in events.user and dispatch_log.approver.

git config user.name = "nikhilsoman"    → events.user = "nikhilsoman", sig = m1
git config user.name = "gaurav-sim"    → events.user = "gaurav-sim", sig = m1
git config user.name = "kunal-sim"     → events.user = "kunal-sim", sig = m1

All three signed by m1 key — same machine, different role simulation.
state.db shows: nikhilsoman built frontend, gaurav-sim reviewed backend.

✓ Team dynamics visible. Cost roll-up per simulated member. No extra infra needed.

Scenario 4 — Real team (Nikhil + Gaurav + Kunal, separate machines)

Nikhil · MacBook (m1)

identity.key → m1 key
git_user: nikhilsoman
workspace: my-product
state.db: primary

Dispatches stories
Approves merges
Exports snapshots

Gaurav · his machine (m2)

identity.key → m2 key
git_user: gaurav
workspace: my-product
state.db: local copy

Imports Nikhil snapshot
Works on stories
Exports back

Kunal · his machine (m3)

identity.key → m3 key
git_user: kunal
workspace: my-product
state.db: local copy

Imports Nikhil snapshot
Works on stories
Exports back
Pre-Tokq sync loop:
Nikhil exports → shares file → Gaurav imports → works → exports → Nikhil imports.
Each import merges events. Gaurav's events carry m2 sig → Nikhil's state.db knows m2 public key (from Gaurav's identity.json, included in export) → verifies signature → trusts attribution.

Trust bootstrap: First import from Gaurav requires Nikhil to confirm his public key fingerprint (one-time, out-of-band — a Slack message: "my fingerprint is ab:cd:ef"). Subsequent imports auto-verify.

Tokq Alpha: All three register. NATS sync replaces export/import entirely. Workspace state converges in real-time. Attribution is cryptographically verified by Tokq cloud, not just locally.
