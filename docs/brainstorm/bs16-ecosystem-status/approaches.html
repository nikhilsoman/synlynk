Three approaches to capability availability

How should the status system be structured?

A

Probe-centric (recommended)

Enhance synlynk probe to produce a full availability snapshot: attach check → version currency → TC-1–4 compliance → verb availability → harness compat baseline. Results land in state.db. synlynk status reads from there — no live checks at status time. Telemetry from real dispatches enriches the harness compat scores over time.

Pros

  • One command refreshes everything
  • Status is always instant (DB read)
  • Telemetry enriches compat scores over time

Cons

  • Probe must be run to get fresh data
  • Harness compat baseline is static until probe runs
B

Separate synlynk compat command

Probe stays lightweight (version + TC checks). A new synlynk compat command owns the matrix: fires synthetic dispatch tests from each harness, measures output fidelity, writes a harness×target matrix to state.db. Status reads from both.

Pros

  • Clean separation of concerns
  • Compat runs independently on demand
  • Synthetic tests catch runtime divergence

Cons

  • Requires all target agents installed to run
  • Synthetic dispatch tests are expensive (~30s+ per agent)
  • Two commands to remember
C

Telemetry-derived (passive)

No active checks. Derive everything from actual dispatch telemetry: attach rate = dispatches in last 24h with exit 0 / total; compat = success rate by verb category per agent; harness experience = job duration + output byte count variance from baseline. Purely passive — no extra work, no synthetic runs.

Pros

  • Zero overhead — uses existing data
  • Reflects actual usage, not synthetic conditions
  • Gets more accurate over time automatically

Cons

  • No signal on day 1 (cold start)
  • Can't distinguish "agent is broken" from "no one dispatched to it recently"
  • Lags real breakage by hours/days
