Capacity layer — three tiers

Tier 1 — Static (v1, in scope)

Curated context window from AGENT_CAPABILITY_BASELINES. Pre-dispatch check: estimated task tokens vs. agent window.

  • Window size in status
  • Dispatch preflight gates on fit
  • No external calls needed

Tier 2 — Plan-aware (v2)

User declares plan tier per agent in .agents/<agent>.json. Pro/free/enterprise limits differ significantly.

  • agy: pro → 1M tokens
  • agy: free → 32K tokens
  • Status shows configured limit

Tier 3 — Dynamic / Aggregator superpower

Detect variable limits: off-peak expansion, regional burst, rate limit headroom. Notify user. Schedule eco-mode jobs into expanded windows.

  • HTTP response header scraping
  • Rate limit header watch
  • Surge window notification
  • Eco mode scheduling

Dispatch modes — capacity-informed routing

eco mode

Queue work for next expanded-capacity or off-peak window. Minimise token cost, maximise context headroom.

  • Holds jobs until surge window detected
  • Prefers large-window agents (agy 1M)
  • Notifies user when window opens
  • Best for: large tasks, non-urgent, cost-sensitive

daily-grind mode (default)

Dispatch to configured default agent immediately. No optimisation. Predictable.

  • Uses per-story agent assignment
  • Preflight gates on capacity fit
  • Escalates to larger window if task won't fit
  • Best for: routine work, known task sizes

perf mode

Dispatch now to best available agent by capability + capacity score. Ignore cost, minimise latency.

  • Routes to highest compat + largest window
  • Parallel dispatch where independent
  • No queue — fires immediately
  • Best for: sprint crunch, blocking work
