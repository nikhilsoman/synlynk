BS-16: Token Budget — Three Dimensions

Context window is one number. Dispatch needs three budgets, each with independent limits and variation axes.

Three independent token budgets

Read / Input

Tokens sent to the model: system prompt, context.md injection, user prompt, tool definitions, prior conversation. Hits a wall silently — model truncates or refuses.

  • Context.md size is the main variable
  • Tool defs add ~2–6K per dispatch
  • Conversation history grows per session
  • Agy 1M → Claude 200K → Codex 128K

Write / Output

Tokens the model can generate per response. Often much smaller than input. If a task needs a long output (full file rewrite, long plan), it hits this first.

  • Claude Max: 32K output
  • Gemini Pro: 8K output (free), 32K (pro)
  • Codex: 16K output
  • Partial output = silently truncated work

Tool Calls

Each tool definition + result consumes input budget. Heavy tool-use tasks (many Read/Glob/Edit calls) can exhaust input before the prompt itself is large.

  • Each tool def: ~200–500 tokens
  • Each result: variable (file reads = large)
  • Agy tool-call loops are the worst offender
  • Tracked separately: avg_tool_calls/job

Variation axes — what changes these limits

Plan level
  • free / pro / max / enterprise
  • Output tokens scale most
  • Rate limits differ sharply
Plan type
  • D2C — consumer subscription
  • Bulk / partner acq
  • API key vs. CLI auth
  • Often undocumented delta
Traffic / time
  • Peak: limits enforced strictly
  • Off-peak: burst headroom
  • Surge windows: 2–5× limits
  • Visible in rate-limit headers
Region / geo
  • API endpoint routing
  • Regional capacity pools
  • Latency proxy for load
  • Tier 3: detect + route

Aggregator edge: Agy has a 1M input window but only 8K output on free tier. synlynk knows this — it won't route a "write 2000 lines of code" task to free-tier Agy even though the input fits. No other dispatch layer models input and output budgets independently.
