Three Approaches to GitHub Projects V2 Integration

All decisions so far: foundational in v0.4 · dynamic field resolution · soft review enforcement

A

Thin Sync Wrapper

Automates the manual GraphQL calls from CLAUDE.md. Synlynk writes TO the board (Agent, Status fields) when agents start work, open PRs, and finish. Board is still GitHub's board; Synlynk just keeps it honest.

New commands: synlynk board connect, synlynk start --issue <n> (sets In Progress + worktree), synlynk pr (sets In Review + suggests reviewer)
Scope: ~3 commands, ~300 lines
What it doesn't solve: Agents in repo A still can't see what's in-flight in repo B

Pros

  • Ships fast
  • Zero new infrastructure
  • Immediately eliminates the copy-paste GraphQL pain

Cons

  • Board is write-only — context is still blind to cross-repo state
  • Doesn't solve the S-037 branch collision problem
B

Board as Bidirectional Context Layer Recommended

Writes to the board AND reads from it. Every synlynk exec / synlynk start queries the board for all In Progress items across registered repos and injects a "Do Not Touch" WIP signal into the agent's context. This is the missing piece that would have prevented S-037.

New commands: all of A, plus board state is injected into .synlynk/context.md on every session start
New context section: ## Live WIP — other agents listing active issues, branches, and agent owners across repos
Scope: ~5 commands + context injection hook, ~600 lines

Pros

  • Solves the actual cross-repo collision problem
  • Board becomes the single live state signal for all agents
  • Complements v0.4 cross-repo standards (standards tell agents what to do; WIP tells them what not to touch)

Cons

  • One extra GitHub API call at every session start
  • Requires all repos to have synlynk board connect run first
C

Full Board Gateway + Dashboard

Synlynk becomes the authoritative task management layer. Adds a local polling daemon that watches the board for changes, a synlynk board status terminal dashboard, and webhook support so the board can trigger Synlynk commands (e.g. assigning an issue auto-starts a worktree).

Adds: polling daemon, synlynk board status rich terminal UI, webhook receiver, board→worktree automation
Scope: ~1200 lines + daemon process
Risk: Daemon adds operational complexity; polling hits GitHub rate limits on large teams

Pros

  • Real-time visibility without manual queries
  • Board drives workflow, not just records it
  • Strong v1.0 Enterprise story

Cons

  • Too heavy for v0.4 — more appropriate as a v0.6+ feature
  • Daemon complexity belongs after the bidirectional layer is proven
