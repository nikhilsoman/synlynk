GitHub Projects V2 — as observed in rxcc

The coordination pattern we're proposing to bring into Synlynk

What rxcc does today (manually, via CLAUDE.md scripts)

Board Schema

  • Columns: Todo → In Progress → In Review → Approved → Done
  • Agent field: Claude / Gemini / Codex
  • Domain label: backend / frontend / infra / testing
  • Status auto-advances on PR open & merge

How Agents Use It

  • GraphQL mutation to set Agent = Claude + Status = In Progress before writing code
  • Label agent:claude added on issue
  • Routing by domain label → right agent picks up the ticket
  • Live issue board as a separate view for production incidents

Pain Points (manual today)

  • Agent must copy-paste a 15-line GraphQL snippet from CLAUDE.md every session
  • No cross-repo visibility — Claude in rxcc can't see Gemini is mid-commit in another repo
  • Board drift when agents forget to update status
  • No automatic agent-routing — relies on domain label existing before session starts

What Synlynk Could Own

  • synlynk start --issue 42 → board sync + worktree + context injection
  • Live WIP signal from board → injected into every agent's context
  • Circular peer-review routing enforced at synlynk pr
  • Cross-repo board aggregation via synlynk board status
