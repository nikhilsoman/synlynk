Section 4 — git-drift

The enforcement artifact: makes the coexistence strategy machine-checkable at commit time

What it does

Input
  • A team manifest: .synlynk/instruction-manifest.json — declares which sections are OS-owned (synlynk), which are skill-pack zones, and which are user-owned
  • The instruction files: CLAUDE.md, .cursorrules, settings.json
Output
  • ✅ Pass — all sections in declared zones, no duplicates, fence intact
  • ⚠️ Warn — skill pack content above the synlynk fence
  • ❌ Block — OS-zone content modified without manifest update; fence removed

The Manifest Schema

{
  "version": "1.0",
  "files": {
    "CLAUDE.md": {
      "fence": {
        "start": "",
        "end": ""
      },
      "zones": {
        "os": "within_fence",        // synlynk owns — block on modification
        "skill_pack": "below_fence", // advisory — warn if above fence
        "user": "below_fence"        // free zone
      },
      "checks": [
        "no_duplicate_headings",
        "fence_intact",
        "skill_pack_below_fence"
      ]
    },
    ".cursorrules": {
      "zones": { "os": "synlynk_block", "skill_pack": "remainder" },
      "checks": ["no_duplicate_rules"]
    }
  }
}

How it installs

Via synlynk init

synlynk init writes the manifest and installs the pre-commit hook automatically. Zero config for new projects.

Standalone (public utility)

pip install git-drift && git-drift install — works in any repo, no synlynk required. The manifest defaults to synlynk conventions but is fully configurable.
