Q8 — How does context reach the agent prompt?

Fixing the global context.md race condition. Three options for the dispatch path.

A

Current — File roundtrip

Write to global file, then read it back

generate_context() called
↓
writes → .synlynk/context.md
(shared, overwritten each dispatch)
↓
reads ← context.md
(may have been overwritten by concurrent job)
↓
embeds into prompts/job-X.md
Race: concurrent dispatch overwrites mid-read
Wasted write + read for every dispatch
Daemon HTTP endpoint shares same file
B
Recommended

In-memory — No dispatch file

generate_context() returns a string; dispatch embeds directly

generate_context() returns string
↓
context_text passed directly
to _format_prompt_for_agent()
↓
embeds into prompts/job-X.md
↓ (separately)
daemon still writes context.md
for HTTP endpoint — no change there
No file write in dispatch path — race eliminated
Minimal code change: add return value to generate_context()
Daemon context.md path unchanged (already lock-guarded)
Can't inspect "what context did job-X get?" after the fact
C

Per-dispatch file

Write context to .synlynk/jobs/<job_id>/context.md

generate_context(output_path=job_context_path)
↓
writes → jobs/job-X/context.md
(isolated per job)
↓
reads ← jobs/job-X/context.md
↓
embeds into prompts/job-X.md
↓ (separately)
daemon still writes context.md
for HTTP endpoint — no change there
Race eliminated — files are per-job
Debuggable: synlynk logs --job X --context can show it
Auditable: injection history per job
More disk writes (one extra file per dispatch)
Larger code change (output_path threading)
Key insight: context.md serves two separate purposes today. The daemon snapshot (HTTP endpoint, watch daemon) should stay at .synlynk/context.md — it's already lock-guarded and works correctly. The dispatch injection doesn't need to be a file at all — it just needs to reach the prompt string. Separating these two paths eliminates the race without changing anything about the daemon.
