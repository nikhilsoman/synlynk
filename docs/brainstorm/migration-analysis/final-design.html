2. synlynk migrate — full flow

Step 1 — Scan

Walk repo tree. Flag any file whose name matches:
roadmap.md, todo.md, memory.md, costs.md,
devlog.md, devlogs/
…that lives outside project-docs/

Also flag agent files at root:
CLAUDE.md, GEMINI.md, AI_INSTRUCTIONS.md
that have substantial content (>100 lines or 3+ unknown sections)

Step 2 — Maturity check

Fresh repo: agent files absent or <30 lines
→ normal init flow, move docs to project-docs/

Evolved repo: agent file >100 lines
OR 3+ sections not in synlynk template
→ show 3-option menu (A/B/C)

3. Semantic section matching — how it works

synlynk maintains an internal map of section coverage signals:

SECTION_SIGNALS = {
  "live issues sop": ["live issue", "live-issue", "sev1", "sev2", "rca"],
  "mid-session anti-amnesia": ["25,000 tokens", "compaction", "checkpoint"],
  "mandatory 4-doc discipline": ["roadmap.md", "devlog", "costs.md", "memory.md"],
  "github projects v2 integration": ["projectId", "updateProjectV2", "PVT_"],
}

For each synlynk section: scan all headers + first 20 lines of each existing section for signals.
2+ signals matched → section considered "covered". Skip append, note as ✓ covered.

4. synlynk start <issue-id> — full flow

1. Read issue
  gh api repos/{owner}/{repo}/issues/{id}
  → title, body, labels

2. Infer agent
  labels → find "agent:*" → strip prefix → look up exec in config
  No agent label → warn + ask: "No agent label found. Which agent? [claude/agy/codex]"

3. Board field discovery
  Check .synlynk/board-cache.json (valid if <7 days old)
  If missing/stale: gh api graphql → discover Status + Agent fields + all options
  Cache result with timestamp

4. Move board
  Set Status = "In Progress" option
  Set Agent = matched option for inferred agent name

5. Build context
  generate_context() → .synlynk/context.md
  Prepend to context.md:
    # Active Issue: #{id} — {title}
    {issue body}

6. Launch
  exec_command([agent_exec])

5. board-cache.json schema

{
  "cached_at": "2026-06-04T10:00:00",
  "project_id": "PVT_kwDOAAUx684BYYIM",
  "status_field_id": "PVTSSF_...",
  "agent_field_id": "PVTSSF_...",
  "status_options": { "In Progress": "47fc9ee4", "Done": "...", ... },
  "agent_options": { "claude": "ecf901a8", "agy": "...", "codex": "..." }
}

6. Error handling

Failure Response
No git remote Warn, skip owner/repo detection, continue
No project_id in config synlynk start fails fast: "Run synlynk init --project-id <id> first"
No agent label on issue Prompt user to choose agent interactively
Board field discovery fails Warn, skip board move, continue to context + launch
Agent exec not found Fail with: "Agent 'claude' exec not found. Is it installed?"

7. CLI surface

synlynk migrate # scan, detect, prompt
synlynk migrate --dry-run # show what would happen, no writes
synlynk start <issue-id> # full flow
synlynk start <issue-id> --dry-run # show plan, don't move board or launch

What this gives rxcc on day 1

Run synlynk migrate → rxcc bootstrapped safely, GH IDs extracted, AGENTS.md created.
Run synlynk start 42 → issue read, board moved, context injected, Claude launched.
No more manual GraphQL. No more copy-pasting context. The board is the source of truth.
