TPM Agent — Design

Lifecycle runner, dispatcher, and cross-story aggregator. Config-driven, same pattern as the Release Agent.

Command Surface
synlynk tpm suggest                    # scan for new plans, surface dispatch proposal (also cron)
synlynk tpm dispatch                   # execute current proposal
synlynk tpm dispatch --story S4        # dispatch a single story
synlynk tpm status                     # show board: all stories + lifecycle stages
synlynk tpm advance --story S4         # manually advance story to next stage (approve-gated stages)
synlynk tpm pause  --story S4          # halt in-flight dispatch for a story
synlynk tpm --install-cron             # register cron job (same pattern as release agent)
Config — project-docs/tpm-agent.json
{
  "tpm_agent": {
    "cron": "0 9 * * 1-5",
    "default_consent": "approve",
    "aggregation_consent": "approve",
    "lifecycle_types": ["software-development", "release"],
    "dispatch_rules": [
      { "if": "task.complexity == 'exploratory'", "assign": "claude" },
      { "if": "task.complexity == 'targeted'",    "assign": "codex"  },
      { "if": "task.complexity == 'scaffold'",    "assign": "agy"    }
    ],
    "capability_overrides": {
      "claude": { "max_parallel": 2 },
      "codex":  { "max_parallel": 4 },
      "agy":    { "max_parallel": 3 }
    },
    "budget": {
      "per_wave_usd": 2.00,
      "per_story_usd": 8.00
    }
  }
}
dispatch_rules
Evaluated top-to-bottom per task. First match wins. Falls back to capability matrix + cost if no rule matches.
capability_overrides
Max parallel slots per agent. TPM won't dispatch more than this many tasks to the same agent concurrently.
budget guards
Estimated cost shown before dispatch. TPM blocks if wave exceeds per_wave_usd and requires explicit approval.

TPM Flow — suggest → dispatch → advance
1
Detect
Scan docs/superpowers/plans/*.md for files newer than last-seen timestamp. Also re-check stories with status=pending in lifecycle_instances (manual seeds).
2
Parse + Wave Assembly
Extract ### Task N blocks. Build dependency graph from depends_on: fields (or infer from ordering). Topological sort → wave groups. Insert rows into tasks table.
3
Agent Assignment
For each task: evaluate dispatch_rules → if no match, score agents via capability_ratings (learned) + AGENT_CAPABILITY_BASELINES (fallback). Pick highest score within budget.
4
Aggregation Check (cross-story)
Scan all stories at same stage. If 2+ have pending Wave 1 tasks with no cross-story deps → surface batch suggestion. Consent: aggregation_consent from config.
5
Human Approves → Dispatch
Print wave plan + estimated cost. Block on [y/N]. On approval: create worktrees, call _run_agent_sync() per task (parallel within wave). Update task status in DB as agents complete.
6
Wave Done → Advance or Block
All tasks in wave done → auto-unlock next wave. Any task BLOCKED → write to sentinel.md, pause story, surface in tpm status. Human resolves → tpm dispatch --story S4 resumes.
7
Lifecycle Stage Advance
All tasks done → check stage's advance trigger. If auto: update lifecycle_instances.stage and notify. If approve: write pending notice to sentinel, wait for tpm advance.
Capability Matrix — how agent assignment works
Agent Strength (baseline) Cost tier Score source
claude Exploratory, multi-file, design judgment high capability_ratings → AGENT_CAPABILITY_BASELINES
codex Targeted, single-file, mechanical implementation medium capability_ratings → AGENT_CAPABILITY_BASELINES
agy Scaffolding, batch, non-interactive setup low capability_ratings → AGENT_CAPABILITY_BASELINES
After each task completes, TPM writes a row to capability_ratings (exit_code, duration, tokens). Over time, the matrix self-improves — if codex keeps failing exploratory tasks, its score for that complexity type drops.
