Lifecycle as First-Class Entity

A lifecycle is a typed, configurable artifact chain. Each stage has a placement point, an artifact it produces, and a set of agent roles that can act on it.

Lifecycle Type Schema
{
  "id": "software-development",
  "label": "Software Dev",
  "stages": [
    {
      "id": "brainstorm",
      "label": "Brainstorm",
      "artifact_out": "brainstorm_session",
      "agent_roles": ["architect"],
      "consent": "approve",
      "advance": "manual"
    },
    {
      "id": "spec",
      "artifact_in": "brainstorm_session",
      "artifact_out": "spec_doc",
      "agent_roles": ["architect", "reviewer"],
      "consent": "approve",
      "advance": "on_file: docs/superpowers/specs/*.md"
    },
    ...
  ]
}
Stage fields
artifact_in — what the stage consumes
artifact_out — what the stage produces
agent_roles — which roles can act here
consent — auto / notify / approve
advance — what triggers next stage
advance triggers
manual — human runs lifecycle advance
on_file: glob — artifact file appears
on_exit_0 — shell command exits clean
on_db: table.status — DB row state

Extensibility
User defines lifecycle in project-docs/lifecycle.json
Each stage maps to known agent roles
Unknown roles = placeholder (attach later)
Lifecycle types stored in .synlynk/lifecycle-types/
Onboarding + Config
synlynk init — prompts "which lifecycle?"
synlynk lifecycle set <type> — switch
synlynk lifecycle status — current stage
synlynk lifecycle advance — manual trigger
Open question
Can multiple lifecycle instances run concurrently?
(e.g. Feature A at Execution stage while Feature B is at Spec stage)
Answer shapes whether lifecycle state is per-project or per-story.
