Model-Aware Capability Scoring + Tokq Memory Units

Brainstorm session — synlynk v0.5.0 + Tokq Alpha

The Gap (Gemini's Proposal)

Current designs score agents at the CLI level — claude-code, gemini-cli, codex. But capability is a function of the underlying model, not the runner. Upgrading claude-3-5-sonnet → claude-3-7-sonnet carries over stale scores silently.

The Bigger Picture (Tokq)

A capability rating row — (model_version, domain, phase, quality) — anchored to a verifiable task is a unit of institutional knowledge. In Tokq, that's a tradeable Memory Unit with provenance, value, and marketplace discoverability.

Three questions to resolve in this session
1
Model version extraction

How do we reliably capture which model executed a task? Self-report, scrape, or require explicit configuration?

2
Domain taxonomy + quality signal

Controlled vocabulary for domains. Automated signal vs. human rating as primary quality source.

3
Capability profile as Tokq Memory Unit

Schema, provenance, valuation. What makes a capability score discoverable and valuable in the marketplace?
