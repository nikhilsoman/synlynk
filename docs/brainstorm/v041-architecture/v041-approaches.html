v0.4.1 — Instruction Reach: Three Approaches

All three solve generation + append. They differ on how the feedback loop works.

A

Section Markers + Manifest

Recommended

Synlynk wraps its contribution in every instruction file with signed markers:
<!-- synlynk:start v0.4.1 --> ... <!-- synlynk:end -->
At init: reads existing file, appends/replaces only the synlynk section. Stores a SHA manifest of what it wrote to .synlynk/instructions.json.

Feedback loop: synlynk instructions status — compares current files against manifest. Reports: synlynk-section unchanged · user-section added 3 lines · external-section detected above synlynk block. Surfaces new sections added by the user or the tool itself for you to review and optionally absorb into the template.

Pros

  • Clean boundary — synlynk never touches user content
  • Idempotent — synlynk init --force only re-writes the synlynk section
  • Feedback loop is precise — knows exactly what changed and where
  • Works for all file formats (markdown, JSON, YAML)

Cons

  • Marker syntax varies per tool (HTML comments vs # comments vs YAML keys)
  • Manifest adds a file to .synlynk/ to maintain
B

Git-Native Drift Detection

Heavier, more powerful

At init, synlynk commits instruction files as a single "synlynk: init instructions" commit and tags the SHAs in .synlynk/instructions.json. Feedback loop uses git diff against those SHAs — any changes to instruction files since that commit are surfaced verbatim.

Feedback loop: synlynk instructions diff — shows a git diff-style output of every instruction file change since synlynk last wrote them. Can be piped to an agent for review: "here's what changed in .cursor/rules since we last generated it — should any of this be absorbed into our template?"

Pros

  • No markers needed — works with any file format natively
  • Full git history available for deeper analysis
  • Diff is human-readable and agent-readable

Cons

  • Requires repo to be git-tracked (instruction files must be committed)
  • Depends on user not rebasing/squashing the init commit
  • Harder to distinguish "user added" vs "tool auto-modified"
C

Sentinel Extension (no new primitives)

Lightest, reuses existing system

Extend the existing sentinel system to watch instruction files. Store hashes of all generated files in .synlynk/telemetry.json. On each synlynk exec, re-hash instruction files and add a INSTRUCTION_DRIFT sentinel event if any changed outside synlynk.

Feedback loop: surfaced in sentinel.md — same severity/ack system already in place. No new commands needed. Integrates with the existing checkpoint flow.

Pros

  • Zero new CLI surface — reuses sentinel + ack
  • Feedback surfaces automatically on every exec, not on-demand
  • Consistent with existing observability model

Cons

  • Hash only — no line-level diff, just "changed / not changed"
  • No way to absorb external changes back into template
  • Sentinel designed for agent behaviour, not file state — conceptual stretch

My recommendation: A + C hybrid. Section markers (A) for clean append/update. Sentinel extension (C) for automatic background detection on every exec — no new command needed to surface drift. The manifest in A gives the diff fidelity that C alone lacks.

Approach B (git-native) is powerful but too fragile — instruction files often aren't committed immediately, and the init commit assumption breaks in real repos.

Take a look and let me know what resonates — or if you want to combine differently.
