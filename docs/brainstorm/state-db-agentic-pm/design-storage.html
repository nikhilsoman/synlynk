Design — Section 1: Storage Layout

What lives where, and how agents reach it from any worktree

Before (current — broken)

repo/
project-docs/  ← tracked in git, branches per worktree, drifts
  memory.md
  roadmap.md
  todo.md
  costs.md
  devlogs/
.synlynk/config.json  ← has project_id already
CLAUDE.md  ← tracked in git ✓
GEMINI.md  ← tracked in git ✓

After (proposed)

repo/ (any worktree)
project-docs/  ← gitignored + removed

.synlynk/
  config.json  ← project_id: "rxcc"
  context.md  ← generated, ephemeral

CLAUDE.md  ← git-tracked (config)
GEMINI.md  ← git-tracked (config)
AGENTS.md  ← git-tracked (config)
~/.synlynk/projects/rxcc/ (shared, machine-local)
state.db  ← SQLite WAL

Tables:
  memory   — key/value + agent + ts
  roadmap  — version, theme, status
  tasks    — todo items, owner, status
  devlog   — append-only entries
  costs    — session rows

One DB, all worktrees read/write here

How any worktree resolves the store

Resolution chain (happens inside synlynk exec)
CWD (any worktree path)
  ↓ walk up to find .synlynk/config.json
read project_id → "rxcc"
  ↓
open ~/.synlynk/projects/rxcc/state.db
  ↓
generate context → .synlynk/context.md
  ↓
spawn agent with context injected

Key invariant

No matter which worktree an agent runs in, it always reads and writes the same state.db. Branching is for code. State never branches.
