# synlynk Vision

**Version:** 2026-05-26  
**Status:** Living document — update at major roadmap inflections  
**Owner:** Nikhil Soman

## Mission

synlynk is the local orchestration layer that makes a developer working with multiple AI tools feel like they have one coherent co-pilot across all of them — no repeated context-setting, no lost state between sessions, no coordination overhead when switching tools or running agents in parallel.

The product earns trust as a **precise local ledger and context compiler** before growing into hosted team sync or enterprise policy. Every model above Solo adds capability on top of a working foundation.

---

## Product Models

| Model | Target User | Projects | Agent Workers | Sync |
|:---|:---|:---|:---|:---|
| **Solo** | 1 dev, any 1 AI tool | Manual docs | No | Local only |
| **Solo+** | 1 dev, GitHub-powered | GitHub Projects v2 | No | Local + GH |
| **Solo+Workers** | 1 dev + 2–3 AI agents | GitHub Projects v2 | Yes (routed by competency) | Local + GH |
| **Team** | 2–10 humans + agents | GH Projects / JIRA / Linear | Yes | Shared (git-backed) |
| **Enterprise** | Org-wide | JIRA / custom | Yes | Hosted sync |

Each model is additive — Solo+ includes all Solo features, Solo+Workers includes all Solo+ features.

### Session protocol (enforced via CLAUDE.md / GEMINI.md / AGENTS.md)

The session protocol is the primary mechanism by which synlynk maintains state continuity. All AI instruction files generated by `synlynk init` enforce the same five-step start and three-step end:

**Session start:**
1. `git config user.name` → identify `@username` for all attribution
2. `synlynk watch status` → start watcher if stopped
3. Read `.synlynk/context.md` → full project state snapshot
4. Check `.synlynk/sentinel.md` → any active alerts
5. Report: last completed task, next active task (+ team activity in team mode)

**During session:**
- Mark tasks `[x]` in todo.md when complete — never delete
- Append decisions to memory.md with `[@username]`
- Run `synlynk checkpoint` at every task boundary
- Log costs to costs.md after significant AI operations

**Session end:**
- Append devlog entry to `project-docs/devlogs/<username>.md`
- `synlynk checkpoint` one final time
- `synlynk status` — include output in closing message

### Project standards baked into Solo

synlynk adopts the following preferences from our established projects (rxcc, pbmr) as defaults embedded in all generated instruction files:

- **Branch naming:** `feat/<desc>` / `fix/<desc>` / `chore/<desc>` — never commit to main/master
- **PR discipline:** One feature per branch; docs updates in separate PRs
- **Co-authored-by trailers:** Every agent commit includes the AI co-author line
- **Attribution:** All memory.md decisions tagged `[@username]`; devlog per user
- **CI paths filter:** All generated GitHub Actions workflows include `paths:` filters — this is the most common agent omission; it is called out explicitly in CLAUDE.md, GEMINI.md, and AGENTS.md
- **Four living docs:** roadmap.md, todo.md, memory.md, costs.md — always in sync; never batched
- **Live issue SOP:** Severity-gated process for production bugs (see `docs/live-issue-handling-sop.md`)

## 2. Solo+ — GitHub Projects v2 Integration

**Target:** A solo developer who wants issue-driven workflow with a managed board — without the overhead of a full team setup.

**Core problem solved:** Solo has project-docs/ as the task system. Solo+ replaces ad-hoc task lists with GitHub Projects v2 as the authoritative board, using the rxcc implementation as the reference model.

### What Solo+ adds

| Feature | Description |
|:---|:---|
| `synlynk init --projects github` | Bootstrap full GH Projects v2 setup (labels, issue templates, workflows, CODEOWNERS) |
| `synlynk assign --issue N` | Read GH issue → determine agent/owner → set board status → post routing comment |
| `synlynk start --issue N` | Claim issue, set board In Progress, create git worktree, inject scoped context, launch AI tool |
| `synlynk pr [--issue N]` | Create PR with `Closes #N`, board auto-moves to In Review on open, Done on merge |
| `synlynk team [status]` | Multi-issue dashboard: active issues, budget, queue |
| Scoped context injection | `start` generates issue-specific context (related tasks + relevant memory only) |
| Agent-state tracking | `.synlynk/agent-state/<agent>.json` records active issue, worktree, start time |
| Worktree management | Per-issue worktrees at `.synlynk/worktrees/<prefix>-<slug>/` |
| Board automation | Four GitHub Actions: agent-label, issue-label, domain-route, stale |

### GitHub Projects v2 bootstrap (reference: rxcc)

`synlynk init --projects github` provisions:

**Label taxonomy** (three orthogonal axes, all applied automatically):

```
type:feature / type:fix / type:chore / type:spec / type:test / type:security
domain:backend / domain:frontend / domain:infra / domain:mobile / domain:testing / domain:design
priority:p0 / priority:p1 / priority:p2 / priority:p3
agent:claude / agent:gemini / agent:codex / agent:human
status:blocked / status:in-review / status:needs-spec / status:stale
```

**Issue templates:** `feature.yml`, `bug.yml`, `chore.yml`, `spec.yml` — form YAML with priority and domain dropdowns that auto-apply labels via `issue-label.yml` workflow.

**Four GitHub Actions workflows:**
1. `agent-label.yml` — reads branch prefix (`/claude/`, `/gemini/`, `/codex/`) → applies `agent:*` label + attribution comment
2. `issue-label.yml` — reads `### Priority` and `### Domain` from issue form → applies labels
3. `domain-route.yml` — on `domain:*` label → posts suggested agent + branch convention + co-author trailer
4. `stale.yml` — marks issues stale at 14 days; closes at 30; exempts p0/blocked/in-review

**Projects v2 board fields:** Sprint (Iteration), Effort (Number), Agent (Single select: Claude / Gemini / Codex / Human / Unassigned)

**Full reference implementation:** `docs/agent-workers/github-pm-playbook.md`

## 3. Solo+Workers — Agent-Workers Model

**Target:** A solo developer running Claude, Gemini, and Codex as parallel first-class contributors, with domain-based routing and performance-tracked allocation.

**Core problem solved:** Picking the right agent for each task is currently an ad-hoc judgment call. Solo+Workers makes it systematic: a competency table drives routing, performance data updates the table, and the tooling enforces the policy.

### What Solo+Workers adds over Solo+

| Feature | Description |
|:---|:---|
| Bootstrap competency config | Default routing table based on empirical rxcc assessment (modifiable) |
| Performance evaluation pipeline | Per-PR scoring → updates competency confidence scores |
| `synlynk handoff --issue N --to <agent>` | Structured agent-to-agent handoff with auto-generated handoff note |
| Worktree locking | `.synlynk/locks/<worktree>.lock` prevents two agents from touching the same files |
| `agent-manifest.md` | Canonical record of which agent owns which issue right now |
| Handoff notes | `.synlynk/handoffs/<issue>-to-<agent>.md` — diff summary, open questions, next steps |
| Cost attribution per agent | costs.md tracks spend by agent; `synlynk team status` shows per-agent totals |
| Auto-checkpoint at handoff | `handoff` runs `checkpoint` before transferring; receiving agent gets fresh context |

### Default competency bootstrap config

Based on empirical assessment across rxcc PRs #218–#250 (see `docs/agent-workers/assessment.md`). Modifiable per project after `synlynk init --workers`.

```json
{
  "competency": {
    "claude": {
      "domains": ["backend", "infra", "auth", "mobile", "design", "spec"],
      "confidence": null,
      "review_gate": "none",
      "notes": "Owns all high-risk code; no confidence ceiling"

    "gemini": {

      "notes": "Clean on isolated UI; risky on auth/JWT payloads — always brief explicitly"
    },
    "codex": {
      "domains": ["testing"],
      "confidence": 6.5,
      "review_gate": "required_before_merge",
      "notes": "Provisional (1 PR). Strong unit test generation; misses CI paths filter"
    }
  },
  "review_overhead_hours": 0.25,
  "reassess_trigger": { "min_prs": 5, "or_after_days": 30 }
}
```

**Known agent weaknesses — must be included in every brief:**

| Agent | Weakness | Mitigation |
|:---|:---|:---|
| Gemini | TypeScript type discipline — misses augmentations | Specify every type augmentation explicitly in spec |
| Gemini | JWT payload completeness — absent fields silently break downstream | List every required JWT field in spec |
| Gemini | Arithmetic edge cases — divide-by-zero on empty/first-user data | Call out edge inputs explicitly |
| Gemini | CI paths filter omission | Remind in spec and GEMINI.md |
| Codex | CI paths filter omission | Remind in spec and AGENTS.md |
| Codex | Semantic error codes — returns misleading HTTP status codes | Specify exact error codes in spec |
| Both | Co-authored-by trailer missing | Enforce via CLAUDE.md / GEMINI.md / AGENTS.md |

### Performance evaluation framework

Source: `docs/agent-workers/perf-tracker.md`

**Metrics collected per PR:**

| Metric | Source |
|:---|:---|
| `loc_added / loc_deleted / loc_net` | `gh pr view --json additions,deletions` |
| `time_to_pr_h` | PR createdAt − issue assigned timestamp |
| `time_to_merge_h` | PR mergedAt − PR createdAt |
| `review_cycles` | Push count after first review request |
| `errors_critical / errors_major / errors_minor` | Review findings |
| `rework_type` | `agent` / `claude` / `none` — who applied fixes |

**Scoring formulas:**

```
quality   = max(0, 10 − (3 × errors_critical) − (2 × errors_major) − errors_minor − (0.5 × review_cycles))
velocity  = (loc_added + loc_deleted) / time_to_pr_h
autonomy  = 2 (clean merge) | 1 (agent fixed on feedback) | 0 (Claude intervention)
composite = (quality × 0.5) + (autonomy × 2.5) + min(velocity/50, 2.5)   # max 10
```

**Policy triggers:**
- Reassess allocation after agent accumulates ≥5 PRs or 30 days
- Demote domain if composite score falls below 5.0 over 3+ PRs
- Promote domain if composite score exceeds 8.0 over 5+ PRs with no critical errors

### Handoff protocol

`synlynk handoff --issue N --to <agent>` generates a structured note at `.synlynk/handoffs/<issue>-to-<agent>.md`:

```markdown
## Handoff: Issue #N → <agent>
**From:** Claude (feat/claude/<slug>)
**Completed:** <list of commits>
**What was built:** <from commit messages>
**Files changed:** <git diff --stat>
**Open questions / known issues:** <from sentinel.md + flagged comments>
**Next steps:** <remaining todo items>
```

This note is injected at the top of the receiving agent's scoped context at `synlynk start`.

## Horizontal Capabilities

These capabilities cut across all models. Implementation depth varies by tier.

### H1. Context Daemon / Live Link

**Current state:** The `synlynk watch` daemon is a Unix background process (`os.fork()`) that polls `project-docs/` every 30 seconds and regenerates `context.md` on any `.md` or `.json` change. This works for Solo — context is fresh at the next session start. The `WatchDaemon.on_change()` method is already stubbed for override (noted in code as "Override in v1.3.0 LCP daemon").

**The problem for Solo+Workers and above:** In multi-agent operation, multiple agents run concurrently in separate worktrees. The current model regenerates a single `context.md` — insufficient when each agent needs a scoped, live view of shared state. An agent writing to `project-docs/` should be visible to a concurrently running agent within seconds, not on next session start.

**Design direction (requires dedicated spec):**

The Live Link is a local IPC layer built on the existing watch daemon foundation. Candidate architecture:
- **Unix domain socket** at `.synlynk/daemon.sock` — daemon listens; agent sessions connect and subscribe to change events
- **Named-pipe pub/sub** — lightweight; one reader per agent session
- **Shared state file with version counter** — agent sessions poll a lightweight `.synlynk/state.json` that is updated atomically on every change; polling interval is much shorter (1–2s) than the current `context.md` regeneration

The extension point is `WatchDaemon.on_change()`. The Live Link is not a new daemon — it is a richer `on_change()` implementation that pushes events to connected agent sessions in addition to regenerating context.

**Constraints to preserve:**
- Daemon remains optional — Solo works fine without it; `synlynk status` and explicit checkpoint remain the primary state surface
- Codex CLI and Composer environments may run in sandboxes where background daemons are unreliable — do not require the daemon in AGENTS.md instructions
- macOS/Linux only for `os.fork()` based daemon; Windows fallback is polling without forking

**Priority:** Design spec required before v0.4.0 work begins on `synlynk start --issue N`, since that command needs to signal running agent sessions when a handoff occurs.

### H2. Environment Coverage

synlynk aims for the widest practical coverage across the environments developers actually use. Priority order:

**Tier 1 — Supported now (via generic `synlynk exec`):**
- Unix terminals: zsh, bash, fish
- AI CLIs: `claude` (Claude Code), `gemini` (Gemini CLI), `codex` (Codex CLI)
- macOS (primary) and Linux

**Tier 2 — First-class integration (per-tool instruction files):**
- Claude Code → `CLAUDE.md` (full session protocol)
- Gemini CLI → `GEMINI.md` (full session protocol)
- Codex CLI / Desktop → `AGENTS.md` (to be implemented; see §Tool Ownership)
- Cursor → `.cursorrules` (minimal, extend to full protocol)
- AntoGravity → to be designed (see §Tool Ownership)

**Tier 3 — IDE terminal / Composer integration:**
- VS Code integrated terminal — works today via generic `exec`; needs `synlynk doctor --vscode`
- JetBrains terminal — same
- Cursor Composer — `.cursorrules` is read; extend to `cursor.json` / Cursor Rules v2 when stable
- Windsurf — needs integration research; likely `.windsurfrules` or equivalent
- GitHub Copilot in editors — research only; no CLI equivalent to `synlynk exec`

**Tier 4 — CI/CD and automation:**
- GitHub Actions — reference workflows provided by `synlynk init --projects github`
- Local task runners — `synlynk codex` non-interactive mode (see Codex tool ownership)
- Container environments — `synlynk doctor` must flag missing PATH/git/project-docs

**Environment coverage principles:**
1. If `synlynk` is not on PATH, `synlynk doctor` should say so clearly with exact fix instructions
2. Do not assume the daemon is running in Composer or CI environments — explicit `status`/`checkpoint` is the safe fallback
3. Shell completions are part of Tier 1 coverage — `synlynk completion zsh|bash|fish`
4. `synlynk doctor` is the primary diagnostic surface; output must be parseable (`--json`) for CI/CD integration

### H3. Cost & Budget Tracking

**Current state:** Manual — AI agents append rows to `project-docs/costs.md`; `check_budgets()` reads the file. Cost rates are not auto-captured from AI CLI output.

**Evolution:**

| Phase | What changes |
|:---|:---|
| v0.3.0 | `synlynk cost add` — structured CLI entry with schema enforcement; replaces free-form markdown |
| v0.4.0 | Provider-specific templates: claude-cost, gemini-cost, codex-cost with correct column formats |
| v0.5.0 (Codex) | JSONL event parsing — capture Codex usage/cost from `codex exec --json` output |
| v0.6.0 | Per-agent and per-issue cost attribution in `synlynk team status` |
| v1.0.0 | One frozen cost schema; `synlynk migrate` upgrades old costs.md files |

**Principle:** Never invent cost values. Record tokens separately when dollars are unavailable. Prefer blank cost + note over fabricated estimate.

### H4. Flatline Sentinel

Detects AI hallucination loops (3 consecutive failures of the same command) and writes alerts to `.synlynk/sentinel.md`. Surfaced in `context.md` and `synlynk status`.

**Evolution:**
- v0.3.0: Include sentinel alert count in `synlynk doctor` output
- v0.5.0: Alert escalation — notify via terminal title after 5 consecutive failures; add `synlynk sentinel clear` command
- Team: Sentinel alerts shared in team telemetry; teammates see active loops

## Tool Ownership Model

Each AI tool is responsible for its own integration design and maintenance within synlynk. This vision document provides directional guidance and feature assignments by model tier. Tool owners drive the implementation, maintain the integration docs, and are responsible for the accuracy of their session protocol.

**Ownership = accountability for:**
- The instruction file(s) generated by `synlynk init` for that tool
- The integration docs in `docs/<tool>/`
- The diagnostic checks in `synlynk doctor --<tool>`
- Compatibility tracking as the tool's CLI/app evolves
- Support playbook for that tool's failure modes

## Projects Integration Strategy

### GitHub Projects v2 — Default for all Solo flavors

GitHub Projects v2 is the default Projects integration for Solo, Solo+, and Solo+Workers. It is the only option that AI agents can interact with natively via `gh` CLI without additional integration glue.

**Why GitHub-only for Solo tiers:**
- `gh` CLI is available on any machine running GitHub workflows
- Agents interact with issues, labels, PRs, and board via `gh api` — no custom integration
- The routing, attribution, and board-update workflows are fully automatable
- Adding a second PM tool (Jira, Linear) creates a coordination gap that breaks agentic workflows

**Reference implementation:** `docs/agent-workers/github-pm-playbook.md` — complete setup recipe including label taxonomy, issue templates, four GitHub Actions workflows, CODEOWNERS, branch protection, and Projects v2 board configuration.

**`synlynk init --projects github` provisions:**
1. Full label taxonomy (type/domain/priority/agent/status axes)
2. Four issue templates (feature, bug, chore, spec) with form YAML
3. PR template with `Closes #N`, domain, test plan, agent attribution
4. Four GitHub Actions workflows
5. CODEOWNERS with agent-domain mapping
6. Branch protection via `gh api`
7. Milestone structure (Private Beta / Public Launch / Backlog)
8. `.synlynk/config.json` `routing` block with domain → agent mapping

### JIRA — Team/Enterprise

Available when `"provider": "jira"` in config. Synlynk connects via JIRA REST API.
- Issue sync (GitHub issue ↔ JIRA ticket)
- Status updates on checkpoint and PR events
- Sprint management available for Team tier
- Required for Enterprise (compliance audit trails)

### Linear — Team alternative

Available when `"provider": "linear"` in config. Linear GraphQL API.
- Faster for small engineering teams than JIRA
- Linear free tier caps at 250 issues — upgrade trigger for growing teams
- Agent interaction is second-class (no Linear CLI) — synlynk provides the bridge

## Open Questions

These require dedicated thinking before being resolved into specs.

### OQ-1: Live Link architecture
The right IPC mechanism for real-time agent-to-agent state sharing. Options: Unix socket, named pipes, atomic shared-state file, or POSIX message queue. Decision affects: daemon complexity, Windows portability, Codex sandbox compatibility, and the `WatchDaemon` extension model.

### OQ-2: Cursor / AntoGravity instruction format
What is the canonical instruction file for each Composer? When does Cursor v2 Rules format stabilize? Does AntoGravity have a project-level config? Until this is answered, `.cursorrules` remains the best effort.

### OQ-3: Codex Desktop App config
`.codex/` project config format is not verified stable. Do not generate `.codex/` files until the format is confirmed — watch Codex CLI changelog and Desktop release notes. Track under `Platform:Codex` label.

### OQ-4: Team sync endpoint
Is the team sync endpoint git-backed (pull-before-write on project-docs/) or a lightweight hosted service? The git-backed approach has no new infra but requires discipline; a service approach enables real-time conflict detection but adds operational overhead. Decision affects Enterprise architecture.

### OQ-5: Linear vs JIRA selection for Team tier
Is Linear or JIRA the default for Team? Linear is faster and developer-friendlier; JIRA is required for many enterprise compliance scenarios. Recommendation: Linear as default for Team; JIRA as upgrade path or Enterprise requirement.

### OQ-6: Automatic vs manual cost capture for Claude/Gemini
Claude Code and Gemini CLI do not emit structured JSONL events the way Codex does. Auto-capturing their token counts requires either output scraping (fragile) or a sidecar process. Is this worth the complexity, or is `synlynk cost add` sufficient for Claude/Gemini while only Codex gets auto-capture via `codex exec --json`?
