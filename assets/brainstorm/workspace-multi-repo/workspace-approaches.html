Workspace Architecture — Three Approaches

Where does workspace state live, and how do repos relate to it?

A

Workspace-only DB ★ Recommended

One state.db per workspace at ~/.synlynk/workspaces/<id>/state.db. Repos are a tag/dimension — a repos table, and every story/event/cost has a repo_id FK. No per-repo DB.

~/.synlynk/workspaces/<workspace_id>/
  state.db        ← everything lives here
  identity.key   ← machine-level keypair
  identity.json

<repo>/.synlynk/
  config.json    ← { workspace_id, repo_id }
  context.md    ← generated (workspace + repo slice)

Pros

  • Single source of truth — one DB to reason about
  • Cross-repo Epic/Story queries are trivial JOINs
  • Solo devs: workspace.db with one repo = identical to current design
  • SQLite WAL handles concurrent reads + sequential agent writes fine

Cons

  • All repos share one SQLite connection pool — minor contention risk at high parallelism
  • Workspace DB path changes (from current ~/.synlynk/projects/<id>/)
B

Federated DBs

Workspace DB for shared state (arcs, phases, epics, identity, constraints, inbox). Per-repo DBs for execution state (stories, events, costs for that repo's work). Cross-repo Epics span both.

~/.synlynk/workspaces/<id>/
  workspace.db   ← arcs, phases, epics, identity, constraints

~/.synlynk/projects/<repo_id>/
  state.db       ← stories, events, costs (this repo only)

<repo>/.synlynk/
  config.json    ← { workspace_id, repo_id }

Pros

  • Per-repo isolation — agents write to their own DB
  • No SQLite write contention between repos

Cons

  • Cross-repo queries require joining across multiple DB files
  • More complex: two DB schemas to maintain
  • Story dispatch logic spans two DBs — harder to reason about
C

Event-log primary + Materialized workspace

Each repo writes to a local append-only event log (.synlynk/events.jsonl). A workspace process periodically materializes all logs into a single workspace state.db. Cross-repo visibility is through the materialized view.

<repo>/.synlynk/
  events.jsonl   ← append-only log (repo writes here)

~/.synlynk/workspaces/<id>/
  state.db       ← materialized view (periodic update)
  identity.key

Pros

  • Zero write contention (each repo writes to its own log)
  • Natural NATS upgrade path (events = messages)
  • Works fully offline per-repo

Cons

  • Workspace state is stale between materializations
  • Cross-repo story coordination requires waiting for next sync cycle
  • Two-layer complexity: log + materialized DB

Why A is recommended

SQLite WAL mode handles the concurrency pattern here comfortably — agents write sequentially within a phase, not 100 concurrent TPS. Cross-repo Epic queries are single-file JOINs. Solo devs get a workspace.db with one repo — structurally identical to the current state.db, just at a different path. Federated DBs (B) solve a contention problem that doesn't materially exist yet. Event-log primary (C) is the right architecture for Tokq/NATS but premature as a local-first design.
