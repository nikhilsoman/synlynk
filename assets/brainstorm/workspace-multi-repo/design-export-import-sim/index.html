The daily sync loop

What the cadence actually looks like
Morning:  synlynk workspace import gaurav-monday.json kunal-monday.json
             → merges their overnight work into your local state.db
             → 2–5 seconds. File size: ~50–200KB for a week of activity.

During day: Work normally. synlynk run, synlynk dispatch, stories complete.
             → all events written to your local state.db, signed with your key.

Evening:  synlynk workspace export --out nikhil-tuesday.json
             → share via Slack DM, email, shared Drive, or team git repo

Reality check: this is basically "git pull at start, git push at end" — but for state, not code.

Full export of a 2-week project: ~80–150KB. --since last-import: ~5–20KB daily.

Honest limitations

It's manual cadence. You have to remember to export and share. Forgetting for 2 days means a bigger merge — more potential conflicts.

Not real-time. Gaurav won't see Nikhil dispatching a story to him until the next import. For a small team with Slack, the "here's my export" message is the notification.

Conflict resolution requires a human. The merge tool surfaces conflicts — it doesn't auto-resolve them. For done_criteria conflicts, that's correct. For status conflicts (both claiming a story?), the rule is: claimed status wins over pending, done wins over in_progress.

This is a bridge, not the destination. Tokq Alpha replaces this entirely with NATS. The export/import design is deliberately simple because it won't last long — it just needs to be good enough for teams who want to use synlynk before Tokq ships.
