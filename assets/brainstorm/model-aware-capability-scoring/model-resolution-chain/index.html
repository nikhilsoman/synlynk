Model Version Resolution Chain

Four-tier fallback — from most reliable to least, with retroactive recovery

1
PRIMARY — Structured header in log

Agent emits # synlynk-meta model=claude-opus-4-8 as first line of every job log. synlynk reads it when the job completes.

Convention injected via CLAUDE.md / GEMINI.md at synlynk init time. Agents we instruct follow it.

reliable
2
FALLBACK 1 — Status / statusline probe

At dispatch time, probe the agent's status endpoint or statusline for the active model. Captures it as a point-in-time snapshot before the job starts. Stored as model_version_at_dispatch.

Claude Code: parse statusline-command.sh line 1
Gemini: gemini --status or env var
Codex: TBD / config file
point-in-time
3
FALLBACK 2 — Explicit config declaration

User-declared in .synlynk/config.json under agent_slots. Used when neither live probe nor header is available. Stale risk: user must update when they change model.

synlynk warns at synlynk status if model_version config is older than 7 days — nudges re-confirmation.

stale risk
4
RECOVERY — Retroactive PR attestation

If model_version = 'unknown' on any story linked to this PR, synlynk pr check blocks merge and posts a comment asking the author agent to attest. Agent responds via synlynk score attest <story-id> --model <version>.

Mid-task /model change handling: if the agent switches model during execution (e.g. /model claude-opus-4-8), the final synlynk-meta header wins. If no header, PR check flags it and the agent is asked: "Which model completed the work?" — retroactive attestation with cryptographic sign-off via Ed25519.
recoverable
Key invariant: A capability rating with model_version = 'unknown' is stored but excluded from routing decisions until resolved. It can still be retroactively promoted once attested — no data is lost.
