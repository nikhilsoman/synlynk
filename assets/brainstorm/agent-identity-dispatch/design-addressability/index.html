Agent Addressability — Where does work land?

Identity says who the agent is. Address says where to send work to reach it.

Two distinct concerns

Dispatch address — machine to agent
How does the daemon/coordinator hand a story to the right agent process?

Needs to work:
· locally (same machine, subprocess)
· across machines (remote agent picking up work)
· cross-network (Tokq era)
Human-agent bridge — approvals & notifications
How does a human get notified, approve, or redirect?

Needs to work:
· approval requests before auto-dispatch
· completion / blocker notifications
· escalation when agent is stuck

Dispatch address options

A

state.db inbox table ★ Recommended for local

Each role has an inbox in state.db. Daemon writes a dispatch_request row. Agent worker (or one-shot synlynk worker --role Builder) polls/watches and picks up. Address = synlynk://<project_id>/roles/builder/inbox resolved locally to state.db.

Pros

  • No new infrastructure — state.db already exists
  • Works with daemon (push) and one-shot (pull)
  • Full audit trail in events table
  • Natural path to NATS (same address scheme, different transport)

Cons

  • Machine-local only until Tokq/NATS
  • Polling adds latency (mitigated: SQLite WAL notify)
B

NATS subject (Tokq-era)

Each role gets a NATS subject: synlynk.<project_id>.builder. Daemon publishes; any subscribed agent picks up. Cross-machine, cross-network out of the box. Already the v1.0 transport.

Pros

  • Cross-machine natively
  • At-most-once or at-least-once delivery semantics
  • Tokq alignment — same transport, same subjects

Cons

  • Requires NATS — not available until v1.0
  • Premature for local-only use
C

Email alias under human account

Each agent role gets an email alias: builder+synlynk@gmail.com or builder@<project>.synlynk.io. Work dispatched cross-machine lands as email; agent polls via Gmail API. Human's email is the routing fabric.

Pros

  • Cross-machine, cross-network today — no NATS needed
  • Human-readable, auditable in existing inbox
  • Integrates with existing notification workflows

Cons

  • Email latency (~seconds to minutes)
  • Requires Gmail API or SMTP setup
  • Not designed for high-frequency dispatch

Human-agent bridge — separate concern, recommended design

Email as the approval + notification channel (not the dispatch channel)
The dispatch address (A or B above) handles machine→agent work handoff.
Email handles the human loop — a different, lower-frequency channel:

Approval request: "Story #42 (refactor auth) is ready to dispatch to Builder/claude.
  Estimated cost: $0.18. Approve? → reply YES / HOLD / REASSIGN"

Completion notice: "Story #42 completed. done_criteria verified. Next: Story #43 queued."

Escalation: "Story #44 blocked — dependency on Story #40 still in_progress for 2h.
  Human review needed."

This makes the human's inbox the audit trail and approval interface.
No new UI needed. Works across machines. Familiar.

Recommended layering

Now (v0.5–v0.7): state.db inbox as dispatch address. Email as human-agent bridge for approvals + notifications.
v1.0+: NATS subject replaces state.db inbox for cross-machine dispatch. Email bridge stays — it's the right channel for human interaction regardless of transport.
