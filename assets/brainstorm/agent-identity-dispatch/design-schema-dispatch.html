id INTEGER PK
agent TEXT     ← "claude-code"
environment TEXT ← "cli"|"desktop"|"ci"

                ["bash","git","gh","edit"]

entitlements

id INTEGER PK
role_id INTEGER FK

                 "git:merge" | "cost_ceiling"
value TEXT      ← "allow"|"block"|"approval"

UNIQUE(role_id, dimension)

inbox

id INTEGER PK

story_id INTEGER FK
status TEXT         ← pending|claimed|done

claimed_at TIMESTAMP

The dispatch address. Agent worker
polls/watches this table.

schedules

name TEXT UNIQUE

Full dispatch flow — all three trigger modes

State transitions from "story pending" to "story running"
── TRIGGER SOURCES ────────────────────────────────────────
[A] story.depends_on satisfied → emit event type="unblocked"
[B] synlynk worker: agent completes → story.status='done' → re-evaluate dependents
[C] schedules.next_run reached → cron tick event
───────────────────────────────────────────────────────────

daemon (or synlynk dispatch --once) wakes on any trigger:
  SELECT stories WHERE status='pending' AND depends_on all done
  FOR each candidate story:
    1. resolve role from story.role_id
    2. check entitlements → decision: auto | approval | hold | reject
    3a. if auto:
        score agent_profiles for this role + domain + quota
        INSERT INTO inbox {role_id, story_id, status='pending'}
        INSERT INTO dispatch_log {decision='auto', routing_reason}
    3b. if approval:
        emit event type='approval_requested'
        send email → wait for reply → on YES → goto 3a
    3c. if reject: emit event type='blocked', block_reason='entitlement'

agent worker (daemon-launched or synlynk worker --role Builder):
  WATCH inbox WHERE role_id=my_role AND status='pending'
  claim story → UPDATE inbox SET status='claimed', claimed_by=my_profile_id
  run: synlynk exec claude (with story context injected)
  on completion → UPDATE stories SET status='done', actual_tokens=N
                → emit event type='completed' → daemon re-evaluates

Identity keypair — where it fits in dispatch

Every dispatch_log row and every completed event is signed with the local Ed25519 private key (.synlynk/identity.key). Signature stored as sig TEXT on the event row. This proves which synlynk instance made each dispatch decision — the audit trail is non-repudiable. At Tokq Alpha, the same signature verifies against the registered public key in the Tokq cloud.
