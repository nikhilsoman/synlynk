Agent Identity & Dispatch — 3 Approaches

The core architectural choice: how does autonomous dispatch work?

A

Pull model — agents poll state.db

No central daemon. Each agent CLI runs a worker loop: check state.db for pending stories assigned to my role + profile, claim one, execute, signal completion, repeat. Trigger sources (cron, git hook, agent completion) all write to state.db; agents read from it on their own schedule.

Pros

  • No persistent daemon — fits single-file, stdlib-only model
  • Agents are independent — one crash doesn't stop others
  • Trivially parallelisable — multiple agents poll the same DB
  • Simple: synlynk worker --role Builder --agent claude

Cons

  • Polling latency — stories don't start instantly
  • Requires agent process to already be running or scheduled externally
  • Harder to enforce entitlements centrally — each agent enforces its own
B

Push model — persistent daemon dispatches ★ Recommended

A watchman-style daemon (already designed in Trio Protocol) owns the dispatch loop. It watches state.db for stories with satisfied dependencies, evaluates entitlements, selects the best-fit agent profile, and launches the agent process. Agents don't need to know about each other — the daemon is the coordinator. Three trigger sources all feed into the same daemon: dependency completion (B), schedule (C), and manual synlynk run (A becomes an API call to the daemon).

Pros

  • Entitlement enforcement is centralised — one place, auditable
  • Near-instant dispatch — no polling gap
  • All three trigger sources (schedule, completion, manual) unified
  • Daemon already designed in Trio Protocol — not starting from scratch

Cons

  • Daemon must stay alive — process management complexity
  • Single point of failure (mitigated: daemon auto-restarts)
  • More complex than pull — harder to debug when silent
C

Hybrid — event-triggered coordinator (stateless)

No persistent process. A lightweight coordinator runs on each triggering event (story completion, cron tick, git hook, manual command) — checks state.db, dispatches ready stories, exits. The OS or a cron job is the persistence layer, not synlynk. Combine with launchd (macOS) or systemd (Linux) for the "always-on" behaviour when needed.

Pros

  • No daemon to manage — OS handles persistence
  • Zero resident memory between events
  • Simpler failure model — each trigger is independent

Cons

  • Latency between event and dispatch (cron granularity = minutes)
  • Requires OS-level integration (launchd/systemd) for always-on
  • Self-chaining (B: agent completes → triggers next) is harder to express

What all three share — the identity model

Regardless of dispatch architecture, agent identity is two-layered:
Role (Architect · Builder · Verifier) → primary entitlement unit — what work can start, what it can touch
Agent Profile (cli · model · competency_scores · environment_capabilities) → fitness function — best-fit selection within a role

An agent earns the right to auto-dispatch a story by: having the role assigned to that story + sufficient competency score for its domain + quota headroom + environment capable of the required actions.
