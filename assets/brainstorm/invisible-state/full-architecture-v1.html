generate_context() — new flow
1 Read state.db → active stories → todo.md view
2 Read notes table → memory.md view
3 Read devlog_entries → last 3 sessions
4 Read source_symbols → architecture block
5 Write context.md → agent reads this
relay sync scope (v0.9.3)
Relay syncs these tables as deltas:
stories · notes · devlog_entries · capability_ratings

NOT synced (local only):
source_symbols · telemetry · autopilot_runs

Conflict resolution: last-write-wins per row for notes. Append-only for devlog_entries and capability_ratings — never conflict.
synlynk migrate — one-time command
$ synlynk migrate
1. Parse project-docs/todo.md → INSERT stories
2. Parse project-docs/memory.md → INSERT notes
3. Parse project-docs/devlogs/*.md → INSERT devlog_entries
4. Parse project-docs/costs.md → already in telemetry
5. git rm -r project-docs/
6. Generate initial views in ~/.synlynk/projects/<hash>/
7. Commit: "chore: migrate project-docs to synlynk state.db"
What the human experiences after migration:
→ "what's next?" — agent reads state.db, answers conversationally
→ "remember we're using postgres" — synlynk memory add "using postgres not mysql" or agent does it inline
→ "show me the roadmap" — synlynk roadmap generates from state.db, prints to terminal
→ job completes → HUD bar expands, collapses after 4s
→ relay syncs teammate's story update → HUD shows "2 stories synced from @teammate"
