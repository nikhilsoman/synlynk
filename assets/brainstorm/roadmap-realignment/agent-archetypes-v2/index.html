What makes SME different from the other archetypes
Trigger model — pull, not push
Maintainers run on a schedule — always watching.
SMEs are consulted — triggered by domain signal:
• Story tagged engg:security → Security SME invoked
• PR touches auth/** → Security SME reviews
• synlynk consult security-sme "is this JWT approach safe?"
• SME can also proactively flag when its domain appears at risk
Level behaviour
Workgroup: codebase-level expert (knows your specific stack)
Team: cross-repo domain reviewer (integration/arch)
Enterprise: standards enforcer (org security policy, compliance)
Community: best-practice curator (published to Tokq marketplace)

→ A "Security SME" sold in the marketplace = packaged security expertise for any project
Platform growth → more archetypes. As synlynk reaches workspace and community levels, new archetypes will emerge naturally — e.g. a Governance archetype (policy enforcement, audit trails) at enterprise level, or a Mentor archetype (onboarding acceleration, knowledge transfer) at community level. The archetype model is open — adding one = new .agents/ config + signal collectors.
