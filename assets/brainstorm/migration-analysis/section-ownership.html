Two approaches to section ownership

A

Section-aware / idempotent ★ recommended

synlynk knows its own section headers. On migrate (and every init): scan for synlynk-owned headers, append only what's missing, never touch the rest.

Checking CLAUDE.md for synlynk sections...
✓ Live Issues SOP already present — skip
+ Mid-Session Anti-Amnesia missing — appending
→ Superpowers, GStack sections: untouched
→ rxcc-specific sections: untouched

Plugin overlap: complementary. Superpowers writes its sections, synlynk writes its sections, rxcc writes its sections. No conflicts — different headers, different content, same file.

B

Section fencing with markers

Wrap synlynk content in explicit markers. Clean boundaries, easy to update or remove programmatically.

<!-- synlynk:start -->
## Live Issues SOP
## Mid-Session Anti-Amnesia Protocol
<!-- synlynk:end -->

Allows synlynk update to replace the fenced block cleanly in future. But HTML comments in Markdown are unusual and may confuse some agents.

Key insight

Superpowers and GStack are complementary — they each own distinct sections. synlynk's sections don't overlap with theirs in practice. Section-aware detection (A) is sufficient for v0.4.0. Fencing (B) is worth adding later if synlynk needs to update its own sections in-place (e.g. a synlynk upgrade-templates command in v1.0).
