Migration UX — What does "merge" mean?

synlynk migrate finds your existing docs. Then what?

Context

rxcc has two types of files to migrate:
1. Agent files (CLAUDE.md, GEMINI.md) — already at repo root, need new sections merged in
2. Project docs (roadmap.md, todo.md, etc.) — may be at root or elsewhere, need moving to project-docs/

A

Move + append missing sections ★ recommended

For project docs: move to project-docs/, preserve content exactly.
For agent files: append missing synlynk sections (Live Issues SOP, Anti-Amnesia etc.) below existing content.
Show a preview of what will be appended before writing.

Terminal output
✓ Found roadmap.md at ./roadmap.md → moving to project-docs/roadmap.md
✓ Found todo.md at ./todo.md → moving to project-docs/todo.md
~ Found CLAUDE.md at ./CLAUDE.md → will append 4 missing sections

Preview additions to CLAUDE.md:
+ ## Live Issues SOP
+ ## Mid-Session Anti-Amnesia Protocol
+ ## Mandatory 4-Doc Discipline
+ ## GitHub Projects v2 Integration

Proceed? [Y/n/s(kip this file)]
B

Move only — no content changes

Move project docs to project-docs/ unchanged. Leave agent files untouched. User manually adds new synlynk sections later (or on next init --force).

Terminal output
✓ Moved ./roadmap.md → project-docs/roadmap.md
✓ Moved ./todo.md → project-docs/todo.md
→ Skipped CLAUDE.md (agent file — edit manually)

Run `synlynk init --force` to update agent instruction files.
C

Full interactive diff

For each file: show a full side-by-side diff (existing vs. what synlynk would write). User picks line by line or section by section what to keep.

Terminal output
--- CLAUDE.md (existing)
+++ CLAUDE.md (synlynk template)
- # rxcc Claude Instructions
+ # synlynk Claude Instructions
...

[k]eep existing [u]se template [m]erge [s]kip

Complex to implement correctly. Risk of data loss if user makes wrong choice.
