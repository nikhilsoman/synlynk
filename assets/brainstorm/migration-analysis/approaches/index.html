synlynk v0.4.0 — Three Approaches

How to structure migration + synlynk start

A

Separate synlynk migrate command ★ recommended

Migration is its own explicit command. synlynk init stays unchanged. User runs synlynk migrate once when ready to adopt the layout.

Pros

  • One-time op gets its own command — clear intent
  • Init stays fast and simple
  • Testable independently
  • User controls timing

Cons

  • User has to discover the command
  • Two commands to bootstrap a new project
B

Migration woven into synlynk init

Before writing any files, init scans the repo, shows what it found, and walks through merge prompts inline. One command does everything.

Pros

  • Single command to fully bootstrap
  • Impossible to skip migration accidentally

Cons

  • Init becomes much longer and interactive
  • Existing users hit migration scan even on re-runs
  • Harder to test in isolation
C

Migration via synlynk doctor

synlynk doctor detects stray files and suggests running migration. Migration itself is still a separate command, but discovery is automatic on every health check.

Pros

  • Most discoverable — doctor surfaces it
  • Clean separation of detection vs. execution

Cons

  • Two separate concepts to explain
  • doctor not yet fully specced

Recommendation

A — separate synlynk migrate. Migration is a one-time repo setup operation, not something init should do on every run. Clear command, clear purpose. We can add a hint to init output later: "Existing docs detected — run synlynk migrate to move them."
