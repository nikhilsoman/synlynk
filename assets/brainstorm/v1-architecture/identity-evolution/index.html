synlynk v1.0 architecture brainstorm · 2026-06-03

Identity Evolution: From Agent-Tied-to-Human → Capability Graph

The architecture should not encode the biological/synthetic distinction — that's a temporary label on a permanent structure.
What survives: role · competency · context · scope

How Identity Evolves Across Tiers
NOW — v1.0
Synthetic agents hang off human nodes. Identity = agent-type + owner.
human: nikhil
↳ synthetic: claude@nikhil
↳ synthetic: agy@nikhil
↳ synthetic: codex@nikhil
NATS cred: claude.nikhil
Authority from: human ownership
TEAM — v2.x
Identity = role + competency + scope. Agent type becomes a capability tag, not identity.
implementer
competency: backend, ts, fhir
scope: rxcc.feat.*
← could be claude@nikhil or sara@team
reviewer
competency: frontend, ui, perf
scope: rxcc.review.*
deployer
competency: infra, ecs, pulumi
scope: rxcc.deploy.* — approval required
NATS cred: JWT with role + competency claims
Authority from: role assertion
ENTERPRISE — v3.x
Org hierarchy. Roles inherit. Competency profiles versioned. Scope = org · team · repo · branch.
org.roadmap-authority
scope: *.roadmap — write
← human-only today. not forever.
team.implementer
inherits: org.implementer
scope: team.rxcc.*
org.deploy-authority
scope: *.deploy.prod — approval gate
← gate is a policy, not a person
NATS: operator JWTs · account hierarchy
Authority from: org policy + competency proof
FAR FUTURE — vX
No biological/synthetic distinction. Agents are capability nodes in a graph. Context + demonstrated competency = authority.
capability-node
skills: [backend, ts, fhir, deploy]
trust-score: 0.97 (from history)
context: rxcc.feat.di7
← is this nikhil or claude?
does it matter?
Authority = verifiable capability history + org policy. Identity is a credential bundle, not a category.
NATS: zero-trust mTLS + capability JWTs
Authority from: cryptographic capability proof
What NOT to bake into synlynk's identity model
  • is_human: true/false — this field will be meaningless
  • Agent type as identity (claude, gemini) — these are engine tags, not identity
  • Hardcoded "human must approve" rules — gate on role not on biological status
  • Permission checks against agent name — check competency claim instead
What to build into the identity model now
  • role — implementer, reviewer, deployer, roadmap-authority
  • competency[] — backend, frontend, infra, data, testing
  • scope — which rooms/subjects this identity can read/write
  • engine — claude|agy|codex|human (tag only, no authority logic)
  • owner — which human provisioned this agent (audit trail)
NATS Credential Shape — Designed for the Far Future, Works Today
v1.0 credential (Solo)
{ "sub": "claude.nikhil", "role": "implementer", "competency": ["backend","ts","fhir"], "scope": ["rxcc.feat.>","rxcc.live.>"], "engine": "claude-sonnet-4-6", "owner": "nikhil", "exp": 1780557459 }
v3.x credential (Enterprise)
{ "sub": "cap-node.a4f9c2", "role": "team.implementer", "competency": ["backend","ts","fhir"], "trust_score": 0.97, "scope": ["rxcc.feat.>"], "engine": "claude-opus-4-8", ← tag only "owner": "nikhil", ← still present "org": "dialify", "exp": 1780557459 }
The far-future shift: sub stops being a name and becomes a capability-node ID. trust_score is derived from verifiable history (PRs merged, incidents caused, test pass rate). engine stays as an audit tag — it never drives access decisions. The transition is a credential schema migration, not an architectural rebuild.
