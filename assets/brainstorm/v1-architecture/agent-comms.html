Inter-Agent Communication Architecture

How do Claude, AntiGravity, and Codex coordinate without talking to each other directly?
This decision shapes the whole autonomy model.

A

GitHub Issue as Message Bus

synlynk exec <issue-id> — the issue IS the unit of work AND the communication channel. Agent reads the issue body + comments for instructions, writes progress as comments, closes when done.

Pros

  • Zero new infrastructure
  • Full audit trail in GitHub
  • Human reads same channel as agents
  • Works today with gh CLI

Cons

  • Noisy issue threads
  • No structured handoff payload
  • No real-time state — agents must poll
B

GitHub Projects v2 Board as Coordination Layer

The board column is the coordination primitive. synlynk start <issue-id> claims the item (moves to In Progress, sets Agent field). The next agent picks up the item when it enters their column (In Review → they are assigned as reviewer).

Pros

  • Matches human workflow exactly
  • Board state = ground truth for all agents
  • No WIP conflicts — claiming is atomic via GH API
  • Human sees full state at a glance

Cons

  • Requires Project ID config per repo
  • GraphQL API calls in every handoff
  • Agent must poll board to discover work
C

Local Handoff File + GitHub as Record

.synlynk/handoff.json is the live coordination state — structured payload passed between agents on the same machine. GitHub issues/PRs are the durable record. synlynk reads handoff.json into context.md at session start.

Pros

  • Structured payload — agent gets exactly what it needs
  • No API calls during coordination
  • Can carry context too large for an issue comment
  • Scoped context slices per agent type

Cons

  • Machine-local only (no remote agent support)
  • Handoff.json = single point of failure if corrupted
  • Not visible in GitHub — human must check locally

My recommendation

B + A combined: GitHub Projects v2 board as the coordination layer (who owns what, what state it's in) with GitHub Issues as the instruction + audit thread (what to build, handoff notes as comments). This maps exactly to how rxcc already works and requires no new infrastructure beyond what synlynk is already building toward.
