synlynk v1.0 architecture brainstorm · 2026-06-03

Future: GitHub-Independent Agent Communication

Three directions — each progressively more powerful, more self-contained, and more efficient than the GitHub primitive.

A

Unix Domain Sockets + MessagePack

synlynk runs a lightweight daemon. Agents communicate via a named socket at ~/.synlynk/agent.sock. Messages are MessagePack-encoded structs — binary JSON equivalent, ~40% smaller, no schema lock-in.

Message shape (conceptual)
{ room: "rxcc/feat-di7", from: "claude", type: "handoff", ts: 1780471000, payload: <msgpack blob> }

Pros

  • Zero latency (kernel-level IPC)
  • Works completely offline
  • ~40% smaller than JSON
  • macOS + Linux native

Cons

  • Machine-local only
  • Daemon must be running
  • Not human-readable without tooling
B

synlynk Context Server — Rooms + Compressed History

synlynk runs a local HTTP+WebSocket server (stdlib only — no deps). Each project gets a room. Each room has a message log stored as line-delimited CBOR or Zstd-compressed JSON. Agents subscribe to rooms relevant to their session. The server generates scoped context slices on demand — no more monolithic context.md.

Pros

  • Rooms = natural context boundaries (no more monolithic context.md)
  • WIP signal solves the S-037 agent collision problem
  • History queryable — agents can ask "what did Gemini decide about X?"
  • Scoped context injection: Claude gets Claude-relevant slice, not everything
  • Cross-repo: rooms span repos on the same machine
  • stdlib HTTP server — zero new deps

Cons

  • Significant new surface area in synlynk
  • Machine-local (remote agents = v2 concern)
  • Server lifecycle management (start/stop/crash)
C

SQLite as the Message Store (no server needed)

No daemon, no sockets. A single ~/.synlynk/state.db SQLite database is the coordination layer. Rooms are tables. Agents write rows when they start/finish work. context.md is generated by querying the DB, not by file-watching. Compressed payloads stored as BLOB columns.

Pros

  • No daemon to manage — SQLite handles concurrency natively
  • Queryable history — SQL is expressive context filtering
  • ACID guarantees — no partial writes, no corruption
  • Single file = trivially backed up, synced via Dropbox/iCloud
  • Python stdlib includes sqlite3 — zero deps
  • BLOB columns for binary payloads (MessagePack/CBOR)

Cons

  • WAL mode needed for concurrent agent writes
  • No real-time push — agents still poll
  • Not human-readable without tooling (or synlynk status)

Recommended evolution path

v1.0: GitHub Issues + Projects v2 (locked) → v1.x: SQLite state store replaces context.md file-watching (Option C, lowest lift) → v2.0: Context Server with rooms (Option B, highest value) — the SQLite store becomes the persistence layer underneath the server.
