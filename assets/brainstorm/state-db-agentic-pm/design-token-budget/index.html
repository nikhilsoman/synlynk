Token Budget, Quota Routing & Platform Sync

Token cost replaces time as the primary project execution constraint

The shift — what replaces story points and sprint velocity

Human PM execution constraints
Story points → effort estimate
Sprint capacity → team hours available
Velocity → points/sprint
Deadline → calendar date
Blocking → waiting on a human

None of these apply to agents.
Agents don't get tired.
Agents don't have capacity.
Agents have quotas.
Agentic PM execution constraints
Token estimate → story cost
Quota headroom → agent availability
Throughput → tokens/quota-period
Critical path → dependency graph
Blocking → unmet dependency or quota

If 3 agents each have 100K hourly quota,
max throughput = 300K tokens/hour.
An epic at 900K tokens = 3 quota-hours
minimum, even with perfect parallelism.

New table — agent_quotas

~/.synlynk/projects/<id>/state.db → agent_quotas
id          INTEGER PK
agent       TEXT       ← "claude" | "gemini" | "codex"
model       TEXT       ← "claude-sonnet-4-6"
quota_type  TEXT       ← "hourly" | "daily" | "weekly" | "monthly"
limit_tokens INTEGER   ← e.g. 100_000 for hourly
used_tokens  INTEGER   ← consumed in current period
headroom     COMPUTED   ← limit_tokens - used_tokens (virtual col)
reset_at     TIMESTAMP  ← when quota resets
updated_at   TIMESTAMP

synlynk exec updates used_tokens after each run via extract_tokens().
synlynk trio status shows live quota headroom per agent.

Routing decision — capability → quota → cost

synlynk run — agent selection for a story
1. story.domain = "api" + story.complexity = "medium"
   → capability_scores WHERE domain='api' ORDER BY score DESC
   → candidates: [claude:0.92, gemini:0.78, codex:0.71]

2. story.estimated_tokens = 40_000
   → agent_quotas WHERE headroom >= 40_000
   → filter out: codex (headroom=12K, reset in 23min)
   → remaining: [claude:88K headroom, gemini:61K headroom]

3. cost comparison for 40K tokens:
   claude-sonnet-4-6 → ~$0.18
   gemini-2.0-flash  → ~$0.04
   capability gap: 0.92 vs 0.78 — within threshold (0.15)
   → ROUTE TO: gemini (capability sufficient, 4.5× cheaper)

story.assigned_agent = "gemini"
story.routing_reason = "capability:ok·quota:61K·cost:$0.04 vs $0.18"
event type="started" content={routing_decision}

New table — external_refs (platform sync)

How Arc/Phase/Epic/Story maps to external platforms
external_refs table:
entity_type TEXT  ← "arc"|"phase"|"epic"|"story"
entity_id   INTEGER
platform    TEXT    ← "github"|"jira"|"linear"|"asana"
external_id TEXT    ← issue #, ticket key, etc.
external_url TEXT
last_synced  TIMESTAMP

Platform mappings:
Arc    → GitHub Project board  | Jira Epic    | Linear Project
Phase  → GitHub Milestone     | Jira Version  | Linear Cycle
Epic   → GitHub Epic label    | Jira Story   | Linear Project
Story  → GitHub Issue        | Jira Sub-task | Linear Issue

state.db is canonical. Platforms are views.
synlynk sync --github pushes; incoming webhooks pull back status changes as events.

What becomes queryable that wasn't before

"Which agent has quota to take this story right now?" — agent_quotas WHERE headroom >= estimated_tokens
"What did Phase v0.5.0 cost total?" — SUM(costs.cost_usd) WHERE phase_id=5
"How accurate are our story estimates?" — AVG(actual_tokens/estimated_tokens) per epic
"What's our throughput ceiling this hour?" — SUM(headroom) across all active agents
"Which stories are blocked by quota, not dependency?" — stories WHERE status='blocked' AND block_reason='quota'
