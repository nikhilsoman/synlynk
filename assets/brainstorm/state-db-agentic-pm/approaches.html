Context Bridge — 3 Approaches

How should synlynk store and serve agent context so all worktrees share the same live state?

A

.git-adjacent SQLite

State lives in .git/synlynk/state.db. Since all worktrees share the same .git dir, every agent reads the same DB with zero config. Gitignored from tracked files. Not cloned — fresh clones start empty and repopulate from first synlynk exec.

Pros

  • Zero config — works instantly
  • Shared by all worktrees automatically
  • No path resolution needed

Cons

  • Lost on fresh clone (no migration path yet)
  • Breaks multi-machine workflows
  • Hidden in .git — not obvious to users
B

~/.synlynk/projects/ home store ★ Recommended

State lives in ~/.synlynk/projects/<project-id>/state.db. All worktrees resolve to the same store via .synlynk/config.json → project_id. Survives re-clones, explicit location, natural home for Tokq sync. Markdown files become generated output, not source of truth.

Pros

  • Shared across all worktrees — no drift possible
  • Survives re-clones and machine migrations
  • Aligns with v0.5.0 SQLite arc
  • Natural Tokq sync target (project_id is the key)
  • Human dump on demand: synlynk context dump

Cons

  • Requires project_id to be set at init
  • Multi-machine sync is manual until Tokq
C

Append-only event log → SQLite projection

Every agent action is an event appended to .git/synlynk/events.jsonl (append-only = no merge conflicts even if git-tracked). SQLite is a read-optimized projection rebuilt from events. Full audit trail and time-travel. This is the v0.9.0 "append-only event log" pulled forward to solve the worktree problem now.

Pros

  • Append-only means no write conflicts ever
  • Full audit trail — who did what, when
  • Time-travel: replay state at any point
  • Natural fit for multi-agent coordination

Cons

  • Most complex to implement now
  • Projection rebuild adds latency
  • Event schema needs careful design upfront

What all three share

Project-docs markdown files (memory.md, roadmap.md, todo.md, devlog) are removed from git tracking entirely. AI templates (CLAUDE.md, GEMINI.md, AGENTS.md) stay in git — they are configuration, not state.
